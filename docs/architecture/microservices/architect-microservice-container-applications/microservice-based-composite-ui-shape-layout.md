# <a name="creating-composite-ui-based-on-microservices"></a><span data-ttu-id="e50b7-104">Criando a interface do usuário composta baseada em microsserviços</span><span class="sxs-lookup"><span data-stu-id="e50b7-104">Creating composite UI based on microservices</span></span>

<span data-ttu-id="e50b7-105">A arquitetura de microserviços geralmente começa com a lógica e os dados de manipulação do lado do servidor, mas, em muitos casos, a interface do usuário ainda é tratada como uma monolítica.</span><span class="sxs-lookup"><span data-stu-id="e50b7-105">Microservices architecture often starts with the server-side handling data and logic, but, in many cases, the UI is still handled as a monolith.</span></span> <span data-ttu-id="e50b7-106">No entanto, uma abordagem mais avançada, chamada de [micro front-ends](https://martinfowler.com/articles/micro-frontends.html), é criar a interface do usuário do aplicativo com base em microserviços também.</span><span class="sxs-lookup"><span data-stu-id="e50b7-106">However, a more advanced approach, called [micro frontends](https://martinfowler.com/articles/micro-frontends.html), is to design your application UI based on microservices as well.</span></span> <span data-ttu-id="e50b7-107">Isso significa ter uma interface do usuário de composição produzida pelos microsserviços, em vez de ter microsserviços no servidor e apenas um aplicativo cliente monolítico consumindo os microsserviços.</span><span class="sxs-lookup"><span data-stu-id="e50b7-107">That means having a composite UI produced by the microservices, instead of having microservices on the server and just a monolithic client app consuming the microservices.</span></span> <span data-ttu-id="e50b7-108">Com essa abordagem, os microsserviços que você criar poderão estar completos com lógica e representação visual.</span><span class="sxs-lookup"><span data-stu-id="e50b7-108">With this approach, the microservices you build can be complete with both logic and visual representation.</span></span>

<span data-ttu-id="e50b7-109">A Figura 4-20 mostra a abordagem mais simples de apenas consumir microsserviços de um aplicativo cliente monolítico.</span><span class="sxs-lookup"><span data-stu-id="e50b7-109">Figure 4-20 shows the simpler approach of just consuming microservices from a monolithic client application.</span></span> <span data-ttu-id="e50b7-110">Obviamente, é possível ter um serviço MVC ASP.NET no meio termo produzindo o HTML e o JavaScript.</span><span class="sxs-lookup"><span data-stu-id="e50b7-110">Of course, you could have an ASP.NET MVC service in between producing the HTML and JavaScript.</span></span> <span data-ttu-id="e50b7-111">A figura é uma simplificação que destaca que você tem uma única interface do usuário do cliente (monolítica) consumindo microsserviços, que se concentram apenas em lógica e nos dados, e não na forma da interface do usuário (HTML e JavaScript).</span><span class="sxs-lookup"><span data-stu-id="e50b7-111">The figure is a simplification that highlights that you have a single (monolithic) client UI consuming the microservices, which just focus on logic and data and not on the UI shape (HTML and JavaScript).</span></span>

<span data-ttu-id="e50b7-115">Em contraste, uma interface do usuário de composição é gerada precisamente e composta pelos próprios microsserviços.</span><span class="sxs-lookup"><span data-stu-id="e50b7-115">In contrast, a composite UI is precisely generated and composed by the microservices themselves.</span></span> <span data-ttu-id="e50b7-116">Alguns do microsserviços promovem a forma visual de áreas específicas da interface do usuário.</span><span class="sxs-lookup"><span data-stu-id="e50b7-116">Some of the microservices drive the visual shape of specific areas of the UI.</span></span> <span data-ttu-id="e50b7-117">A principal diferença é que você tem componentes de interface do usuário do cliente (classes TypeScript, por exemplo) com base em modelos, sendo que o ViewModel da interface do usuário de modelagem de dados para esses modelos é obtido de cada microsserviço.</span><span class="sxs-lookup"><span data-stu-id="e50b7-117">The key difference is that you have client UI components (TypeScript classes, for example) based on templates, and the data-shaping-UI ViewModel for those templates comes from each microservice.</span></span>

<span data-ttu-id="e50b7-118">Em tempo de inicialização do aplicativo cliente, cada um dos componentes de interface do usuário do cliente (classes TypeScript, por exemplo) se registra com um microsserviço de infraestrutura capaz de fornecer ViewModels para um determinado cenário.</span><span class="sxs-lookup"><span data-stu-id="e50b7-118">At client application start-up time, each of the client UI components (TypeScript classes, for example) registers itself with an infrastructure microservice capable of providing ViewModels for a given scenario.</span></span> <span data-ttu-id="e50b7-119">Se o microsserviço alterar a forma, a interface do usuário também será alterada.</span><span class="sxs-lookup"><span data-stu-id="e50b7-119">If the microservice changes the shape, the UI changes also.</span></span>

<span data-ttu-id="e50b7-128">Uma abordagem de interface do usuário composta que é controlada por microsserviços pode ser mais desafiadora ou menos, dependendo de quais tecnologias de interface do usuário estão sendo usadas.</span><span class="sxs-lookup"><span data-stu-id="e50b7-128">A composite UI approach that's driven by microservices can be more challenging or less so, depending on what UI technologies you're using.</span></span> <span data-ttu-id="e50b7-129">Por exemplo, você não usará as mesmas técnicas para criar um aplicativo Web tradicional que você usa para criar um SPA ou para o aplicativo móvel nativo (como acontece durante o desenvolvimento de aplicativos Xamarin, que podem ser mais desafiadores para essa abordagem).</span><span class="sxs-lookup"><span data-stu-id="e50b7-129">For instance, you won't use the same techniques for building a traditional web application that you use for building an SPA or for native mobile app (as when developing Xamarin apps, which can be more challenging for this approach).</span></span>

<span data-ttu-id="e50b7-130">O aplicativo de exemplo [eShopOnContainers](https://aka.ms/MicroservicesArchitecture) usa a abordagem de interface do usuário monolítica por vários motivos.</span><span class="sxs-lookup"><span data-stu-id="e50b7-130">The [eShopOnContainers](https://aka.ms/MicroservicesArchitecture) sample application uses the monolithic UI approach for multiple reasons.</span></span> <span data-ttu-id="e50b7-131">Primeiro, é uma introdução a microsserviços e contêineres.</span><span class="sxs-lookup"><span data-stu-id="e50b7-131">First, it's an introduction to microservices and containers.</span></span> <span data-ttu-id="e50b7-132">Uma interface do usuário de composição é mais avançada, mas também requer mais complexidade ao criar e desenvolver a interface do usuário.</span><span class="sxs-lookup"><span data-stu-id="e50b7-132">A composite UI is more advanced but also requires further complexity when designing and developing the UI.</span></span> <span data-ttu-id="e50b7-133">Em segundo lugar, o eShopOnContainers também oferece um aplicativo móvel nativo baseado no Xamarin, que tornaria isso mais complexo no lado C\# do cliente.</span><span class="sxs-lookup"><span data-stu-id="e50b7-133">Second, eShopOnContainers also provides a native mobile app based on Xamarin, which would make it more complex on the client C\# side.</span></span>
