## <a name="circuit-breaker-pattern"></a><span data-ttu-id="358e0-146">Padrão de disjuntor</span><span class="sxs-lookup"><span data-stu-id="358e0-146">Circuit breaker pattern</span></span>

<span data-ttu-id="358e0-147">Embora o padrão de repetição possa ajudar a recuperar uma solicitação confusas em uma falha parcial, há situações em que as falhas podem ser causadas por eventos inesperados que exigirão períodos de tempo mais longos para resolver.</span><span class="sxs-lookup"><span data-stu-id="358e0-147">While the retry pattern can help salvage a request entangled in a partial failure, there are situations where failures can be caused by unanticipated events that will require longer periods of time to resolve.</span></span> <span data-ttu-id="358e0-148">Essas falhas podem variar em termos de gravidade de uma perda parcial de conectividade até a falha completa de um serviço.</span><span class="sxs-lookup"><span data-stu-id="358e0-148">These faults can range in severity from a partial loss of connectivity to the complete failure of a service.</span></span> <span data-ttu-id="358e0-149">Nessas situações, é inútil que um aplicativo repita continuamente uma operação que provavelmente não terá sucesso.</span><span class="sxs-lookup"><span data-stu-id="358e0-149">In these situations, it's pointless for an application to continually retry an operation that is unlikely to succeed.</span></span>

<span data-ttu-id="358e0-150">Para piorar as coisas, a execução de operações contínuas de repetição em um serviço não responsivo pode movê-lo para um cenário de negação de serviço autoimposto, no qual você inunda o serviço com chamadas contínuas esgotando recursos como memória, threads e banco de dados conexões, causando falha em partes não relacionadas do sistema que usam os mesmos recursos.</span><span class="sxs-lookup"><span data-stu-id="358e0-150">To make things worse, executing continual retry operations on a non-responsive service can move you into a self-imposed denial of service scenario where you flood your service with continual calls exhausting resources such as memory, threads and database connections, causing failure in unrelated parts of the system that use the same resources.</span></span>

<span data-ttu-id="358e0-151">Nessas situações, seria preferível para a operação falhar imediatamente e tentar invocar o serviço apenas se for provável que tenha êxito.</span><span class="sxs-lookup"><span data-stu-id="358e0-151">In these situations, it would be preferable for the operation to fail immediately and only attempt to invoke the service if it's likely to succeed.</span></span>

<span data-ttu-id="358e0-152">O [padrão de disjuntor](https://docs.microsoft.com/azure/architecture/patterns/circuit-breaker) pode impedir que um aplicativo tente executar repetidamente uma operação que provavelmente falhará.</span><span class="sxs-lookup"><span data-stu-id="358e0-152">The [Circuit Breaker pattern](https://docs.microsoft.com/azure/architecture/patterns/circuit-breaker) can prevent an application from repeatedly trying to execute an operation that's likely to fail.</span></span> <span data-ttu-id="358e0-153">Ele também monitora o aplicativo com uma chamada de avaliação periódica para determinar se a falha foi resolvida.</span><span class="sxs-lookup"><span data-stu-id="358e0-153">It also monitors the application with a periodic trial call to determine whether the fault has resolved.</span></span> <span data-ttu-id="358e0-154">A Figura 6-5 mostra o padrão de disjuntor em ação.</span><span class="sxs-lookup"><span data-stu-id="358e0-154">Figure 6-5 shows the Circuit Breaker pattern in action.</span></span>
