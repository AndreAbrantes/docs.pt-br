# <a name="conclusion"></a>Conclusão

Os principais argumentos a seguir são as conclusões mais importantes deste guia.

**Benefícios do uso sem servidor.** As soluções sem servidor fornecem o benefício importante da economia de custos porque o servidor é implementado em um modelo de pagamento por uso. Sem servidor torna possível dimensionar, testar e implantar de forma independente componentes individuais do seu aplicativo. O servidor não é exclusivamente adequado para implementar arquiteturas de microserviços e integra-se totalmente a um pipeline DevOps.

**Código como uma unidade de implantação.** Sem servidor abstrai o hardware, o sistema operacional e o tempo de execução fora do aplicativo. O sem servidor permite focar na lógica de negócios no código como a unidade de implantação.

**Gatilhos e associações.** A integração sem servidor facilita o armazenamento, as APIs e outros recursos de nuvem. Azure Functions fornece gatilhos para executar código e associações para interagir com recursos.

**Microsserviços.** A arquitetura de microserviços está se tornando a abordagem preferida para aplicativos de missão crítica distribuídos e grandes ou complexos que se baseiam em vários subsistemas independentes na forma de serviços autônomos. Em uma arquitetura baseada em microserviço, o aplicativo é criado como uma coleção de serviços que podem ser desenvolvidos, testados, com controle de versão, implantados e dimensionados de forma independente. Sem servidor é uma arquitetura bem adequada para a criação desses serviços.

**Plataformas sem servidor.** Sem servidor não é apenas o código. As plataformas que oferecem suporte a arquiteturas sem servidor incluem fluxos de trabalho e orquestração de servidor, serviços de mensagens e eventos sem servidor e bancos de dados sem servidor.

**Desafios sem servidor.** Sem servidor apresenta os desafios relacionados ao desenvolvimento de aplicativos distribuídos, como modelos de dados fragmentados e independentes, resiliência, controle de versão e descoberta de serviços. Sem servidor, talvez não seja ideal para processos ou componentes de longa execução que se beneficiem de um acoplamento mais rígido.

**Sem servidor como uma ferramenta, não a caixa de ferramentas.** Sem servidor não é a solução exclusiva para a arquitetura do aplicativo. É uma ferramenta que pode ser aproveitada como parte de um aplicativo híbrido que pode conter camadas tradicionais, back-ends monolíticos e contêineres. Sem servidor pode ser usado para aprimorar soluções existentes e não é uma abordagem de tudo ou nada para o desenvolvimento de aplicativos.
