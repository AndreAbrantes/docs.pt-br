# <a name="conclusion"></a>Conclusão

As principais lições a seguir estão as conclusões mais importantes deste guia.

**Benefícios do uso sem servidor.** Soluções sem servidor fornecem benefícios importantes para economia de custos, pois sem servidor é implementado em um modelo de pagamento por uso. Serverless torna possível dimensionar independentemente, testar e implantar os componentes individuais do seu aplicativo. Sem servidor é especialmente adequada para implementar arquiteturas de microsserviços e integra-se totalmente em um pipeline de DevOps.

**O código como uma unidade de implantação.** Serverless abstrai o hardware, o sistema operacional e o tempo de execução para fora do aplicativo. Sem servidor permite se concentrar na lógica de negócios em código como a unidade de implantação.

**Gatilhos e associações.** Serverless facilita a integração com o armazenamento, APIs e outros recursos de nuvem. O Azure Functions oferece gatilhos para executar o código e as associações para interagir com os recursos.

**Microsserviços.** A arquitetura de microsserviços está se tornando a abordagem preferencial para aplicativos críticos distribuídos grandes ou complexos e com base em diversos subsistemas independentes na forma de serviços autônomos. Em uma arquitetura baseada em microsserviço, o aplicativo é compilado como uma coleção de serviços que podem ser desenvolvidos, testados com controle de versão, implantados e dimensionados de forma independente. Sem servidor é uma arquitetura adequada para a criação desses serviços.

**Plataformas sem servidor.** Sem servidor não é apenas sobre o código. Plataformas que dão suporte a arquiteturas sem servidor incluem fluxos de trabalho sem servidor e orquestração, sem servidor de mensagens e serviços de eventos e os bancos de dados sem servidor.

**Desafios sem servidor.** Sem servidor apresenta desafios relacionados ao desenvolvimento de aplicativos distribuídos, tais como fragmentado e modelos de dados independente, resiliência, controle de versão e descoberta de serviço. Sem servidor pode não ser ideal para processos de longa execução ou componentes que se beneficiam de acoplamento.

**Sem servidor como uma ferramenta, não a caixa de ferramentas.** Sem servidor é não a solução exclusiva para a arquitetura do aplicativo. É uma ferramenta que pode ser utilizada como parte de um aplicativo híbrido que pode conter camadas tradicionais, monólito back-ends e contêineres. Sem servidor pode ser usado para aprimorar soluções existentes e não é uma abordagem tudo ou nada para desenvolvimento de aplicativos.
