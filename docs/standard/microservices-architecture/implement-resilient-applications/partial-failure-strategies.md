# <a name="strategies-for-handling-partial-failure"></a><span data-ttu-id="81607-103">Estratégias para tratar falhas parciais</span><span class="sxs-lookup"><span data-stu-id="81607-103">Strategies for handling partial failure</span></span>

<span data-ttu-id="81607-104">As estratégias para lidar com falhas parciais incluem o seguinte.</span><span class="sxs-lookup"><span data-stu-id="81607-104">Strategies for dealing with partial failures include the following.</span></span>

<span data-ttu-id="81607-105">**Usar a comunicação assíncrona (por exemplo, comunicação baseada em mensagens) entre microsserviços internos**.</span><span class="sxs-lookup"><span data-stu-id="81607-105">**Use asynchronous communication (for example, message-based communication) across internal microservices**.</span></span> <span data-ttu-id="81607-106">É altamente recomendável não criar longas cadeias de chamadas HTTP síncronas entre os microsserviços internos, porque esse design incorreto eventualmente se tornará a principal causa de interrupções incorretas.</span><span class="sxs-lookup"><span data-stu-id="81607-106">It is highly advisable not to create long chains of synchronous HTTP calls across the internal microservices because that incorrect design will eventually become the main cause of bad outages.</span></span> <span data-ttu-id="81607-107">Pelo contrário, exceto pelas comunicações de front-end entre os aplicativos cliente e o primeiro nível de microsserviços ou Gateways de API refinados, é recomendável usar apenas comunicação assíncrona (com base em mensagem) uma vez após o ciclo de resposta/solicitação inicial, entre os microsserviços internos.</span><span class="sxs-lookup"><span data-stu-id="81607-107">On the contrary, except for the front-end communications between the client applications and the first level of microservices or fine-grained API Gateways, it is recommended to use only asynchronous (message-based) communication once past the initial request/response cycle, across the internal microservices.</span></span> <span data-ttu-id="81607-108">Consistência eventual e arquiteturas orientadas a eventos ajudarão a minimizar os efeitos de ondulação.</span><span class="sxs-lookup"><span data-stu-id="81607-108">Eventual consistency and event-driven architectures will help to minimize ripple effects.</span></span> <span data-ttu-id="81607-109">Essas abordagens impõem um nível mais alto de autonomia do microsserviço e, portanto, previnem contra o problema observado aqui.</span><span class="sxs-lookup"><span data-stu-id="81607-109">These approaches enforce a higher level of microservice autonomy and therefore prevent against the problem noted here.</span></span>

<span data-ttu-id="81607-110">**Usar novas tentativas com retirada exponencial**.</span><span class="sxs-lookup"><span data-stu-id="81607-110">**Use retries with exponential backoff**.</span></span> <span data-ttu-id="81607-111">Essa técnica ajuda a evitar falhas curtas e intermitentes executando novas tentativas de chamada um determinado número de vezes, caso o serviço não tivesse estado disponível apenas por um curto período de tempo.</span><span class="sxs-lookup"><span data-stu-id="81607-111">This technique helps to avoid short and intermittent failures by performing call retries a certain number of times, in case the service was not available only for a short time.</span></span> <span data-ttu-id="81607-112">Isso pode ocorrer devido a problemas de rede intermitentes ou quando um microsserviço/contêiner é movido para um nó diferente em um cluster.</span><span class="sxs-lookup"><span data-stu-id="81607-112">This might occur due to intermittent network issues or when a microservice/container is moved to a different node in a cluster.</span></span> <span data-ttu-id="81607-113">No entanto, se essas novas tentativas não foram criadas corretamente com disjuntores, os efeitos de ondulação poderão ser agravados e, em último caso, poderá até haver uma [DoS (Negação de Serviço)](https://en.wikipedia.org/wiki/Denial-of-service_attack).</span><span class="sxs-lookup"><span data-stu-id="81607-113">However, if these retries are not designed properly with circuit breakers, it can aggravate the ripple effects, ultimately even causing a [Denial of Service (DoS)](https://en.wikipedia.org/wiki/Denial-of-service_attack).</span></span>

<span data-ttu-id="81607-114">**Solução alternativa para tempos limite de rede**.</span><span class="sxs-lookup"><span data-stu-id="81607-114">**Work around network timeouts**.</span></span> <span data-ttu-id="81607-115">Em geral, os clientes devem ser criados para não serem bloqueados indefinidamente e para sempre usar tempos limite ao aguardar uma resposta.</span><span class="sxs-lookup"><span data-stu-id="81607-115">In general, clients should be designed not to block indefinitely and to always use timeouts when waiting for a response.</span></span> <span data-ttu-id="81607-116">Usar tempos limite garante que os recursos nunca fiquem bloqueados indefinidamente.</span><span class="sxs-lookup"><span data-stu-id="81607-116">Using timeouts ensures that resources are never tied up indefinitely.</span></span>

<span data-ttu-id="81607-117">**Usar o padrão de disjuntor**.</span><span class="sxs-lookup"><span data-stu-id="81607-117">**Use the Circuit Breaker pattern**.</span></span> <span data-ttu-id="81607-118">Nessa abordagem, o processo do cliente rastreia o número de solicitações com falha.</span><span class="sxs-lookup"><span data-stu-id="81607-118">In this approach, the client process tracks the number of failed requests.</span></span> <span data-ttu-id="81607-119">Se a taxa de erro exceder um limite configurado, um "disjuntor" viajará para que outras tentativas falhem imediatamente.</span><span class="sxs-lookup"><span data-stu-id="81607-119">If the error rate exceeds a configured limit, a “circuit breaker” trips so that further attempts fail immediately.</span></span> <span data-ttu-id="81607-120">(Se um alto número de solicitações estão apresentando falha, isso sugere que o serviço não está disponível e que enviar solicitações é inútil.) Após um período de tempo limite, o cliente deverá tentar novamente e, se as novas solicitações forem bem-sucedidas, feche o disjuntor.</span><span class="sxs-lookup"><span data-stu-id="81607-120">(If a large number of requests are failing, that suggests the service is unavailable and that sending requests is pointless.) After a timeout period, the client should try again and, if the new requests are successful, close the circuit breaker.</span></span>

<span data-ttu-id="81607-121">**Fornecer fallbacks**.</span><span class="sxs-lookup"><span data-stu-id="81607-121">**Provide fallbacks**.</span></span> <span data-ttu-id="81607-122">Nessa abordagem, o processo de cliente executa a lógica de fallback quando uma solicitação falha, como retornar dados armazenados em cache ou um valor padrão.</span><span class="sxs-lookup"><span data-stu-id="81607-122">In this approach, the client process performs fallback logic when a request fails, such as returning cached data or a default value.</span></span> <span data-ttu-id="81607-123">Essa é uma abordagem adequada para consultas e é mais complexa para atualizações ou comandos.</span><span class="sxs-lookup"><span data-stu-id="81607-123">This is an approach suitable for queries, and is more complex for updates or commands.</span></span>

<span data-ttu-id="81607-124">**Limitar o número de solicitações na fila**.</span><span class="sxs-lookup"><span data-stu-id="81607-124">**Limit the number of queued requests**.</span></span> <span data-ttu-id="81607-125">Os clientes também devem impor um limite superior no número de solicitações pendentes que um microsserviço cliente pode enviar para um serviço específico.</span><span class="sxs-lookup"><span data-stu-id="81607-125">Clients should also impose an upper bound on the number of outstanding requests that a client microservice can send to a particular service.</span></span> <span data-ttu-id="81607-126">Se o limite tiver sido atingido, provavelmente será ineficaz fazer solicitações adicionais, e essas tentativas devem falhar imediatamente.</span><span class="sxs-lookup"><span data-stu-id="81607-126">If the limit has been reached, it is probably pointless to make additional requests, and those attempts should fail immediately.</span></span> <span data-ttu-id="81607-127">Em termos de implementação, a política [Isolamento bulkhead](https://github.com/App-vNext/Polly/wiki/Bulkhead) da Polly pode ser usada para atender a esse requisito.</span><span class="sxs-lookup"><span data-stu-id="81607-127">In terms of implementation, the Polly [Bulkhead Isolation](https://github.com/App-vNext/Polly/wiki/Bulkhead) policy can be used to fulfil this requirement.</span></span> <span data-ttu-id="81607-128">Essa abordagem é essencialmente uma restrição de paralelização com <xref:System.Threading.SemaphoreSlim> como a implementação.</span><span class="sxs-lookup"><span data-stu-id="81607-128">This approach is essentially a parallelization throttle with <xref:System.Threading.SemaphoreSlim> as the implementation.</span></span> <span data-ttu-id="81607-129">Ela também permite uma "fila" fora do bulkhead.</span><span class="sxs-lookup"><span data-stu-id="81607-129">It also permits a "queue" outside the bulkhead.</span></span> <span data-ttu-id="81607-130">É possível lançar proativamente uma carga excessiva mesmo antes da execução (por exemplo, devido à capacidade ser considerada cheia).</span><span class="sxs-lookup"><span data-stu-id="81607-130">You can proactively shed excess load even before execution (for example, because capacity is deemed full).</span></span> <span data-ttu-id="81607-131">Isso torna sua resposta a determinados cenários de falha mais rápida do que um disjuntor seria, uma vez que o disjuntor aguarda as falhas.</span><span class="sxs-lookup"><span data-stu-id="81607-131">This makes its response to certain failure scenarios faster than a circuit breaker would be, since the circuit breaker waits for the failures.</span></span> <span data-ttu-id="81607-132">O objeto BulkheadPolicy na Polly expõe o quão cheios o bulkhead e a fila estão e oferece eventos em estouro para que também possam ser usados para promover a colocação em escala horizontal automatizada.</span><span class="sxs-lookup"><span data-stu-id="81607-132">The BulkheadPolicy object in Polly exposes how full the bulkhead and queue are, and offers events on overflow so can also be used to drive automated horizontal scaling.</span></span>
