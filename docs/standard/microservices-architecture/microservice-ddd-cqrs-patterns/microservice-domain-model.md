# <a name="designing-a-microservice-domain-model"></a><span data-ttu-id="212aa-104">Criando um modelo de domínio de microsserviço</span><span class="sxs-lookup"><span data-stu-id="212aa-104">Designing a microservice domain model</span></span>

<span data-ttu-id="212aa-105">*Definir um modelo de domínio avançado para cada microsserviço de negócios ou Contexto Limitado*</span><span class="sxs-lookup"><span data-stu-id="212aa-105">*Define one rich domain model for each business microservice or Bounded Context*</span></span>

<span data-ttu-id="212aa-106">Sua meta é criar um único modelo de domínio coeso para cada microsserviço de negócios ou BC (Contexto Limitado).</span><span class="sxs-lookup"><span data-stu-id="212aa-106">Your goal is to create a single cohesive domain model for each business microservice or Bounded Context (BC).</span></span> <span data-ttu-id="212aa-107">No entanto, tenha em mente que um microsserviço de BC ou de continuidade de negócios, às vezes, pode ser composto por vários serviços físicos que compartilham um único modelo de domínio.</span><span class="sxs-lookup"><span data-stu-id="212aa-107">Keep in mind, however, that a BC or business microservice could sometimes be composed of several physical services that share a single domain model.</span></span> <span data-ttu-id="212aa-108">O modelo de domínio precisa capturar as regras, o comportamento, a linguagem de negócios e as restrições do único Contexto Limitado ou microsserviço de negócios que ele representa.</span><span class="sxs-lookup"><span data-stu-id="212aa-108">The domain model must capture the rules, behavior, business language, and constraints of the single Bounded Context or business microservice that it represents.</span></span>

## <a name="the-domain-entity-pattern"></a><span data-ttu-id="212aa-109">O padrão de entidade de domínio</span><span class="sxs-lookup"><span data-stu-id="212aa-109">The Domain Entity pattern</span></span>

<span data-ttu-id="212aa-110">As entidades representam objetos de domínio e são definidas principalmente pela identidade, a continuidade e a persistência ao longo do tempo e não apenas pelos atributos que as compõem.</span><span class="sxs-lookup"><span data-stu-id="212aa-110">Entities represent domain objects and are primarily defined by their identity, continuity, and persistence over time, and not only by the attributes that comprise them.</span></span> <span data-ttu-id="212aa-111">Como Eric Evans afirma, "um objeto definido principalmente pela sua identidade é chamado de entidade”.</span><span class="sxs-lookup"><span data-stu-id="212aa-111">As Eric Evans says, “an object primarily defined by its identity is called an Entity.”</span></span> <span data-ttu-id="212aa-112">As entidades são muito importantes no modelo de domínio, pois elas são a base para um modelo.</span><span class="sxs-lookup"><span data-stu-id="212aa-112">Entities are very important in the domain model, since they are the base for a model.</span></span> <span data-ttu-id="212aa-113">Portanto, você deve identificá-las e criá-las com cuidado.</span><span class="sxs-lookup"><span data-stu-id="212aa-113">Therefore, you should identify and design them carefully.</span></span>

<span data-ttu-id="212aa-114">*A identidade de uma entidade pode cruzar vários microsserviços ou contextos limitados.*</span><span class="sxs-lookup"><span data-stu-id="212aa-114">*An entity’s identity can cross multiple microservices or Bounded Contexts.*</span></span>

<span data-ttu-id="212aa-115">A mesma identidade (mas não a mesma entidade) pode ser modelada em vários Contextos Limitados ou microsserviços.</span><span class="sxs-lookup"><span data-stu-id="212aa-115">The same identity (though not the same entity) can be modeled across multiple Bounded Contexts or microservices.</span></span> <span data-ttu-id="212aa-116">No entanto, isso não significa que a mesma entidade, com os mesmos atributos e a mesma lógica, possa ser implementada em vários Contextos Limitados.</span><span class="sxs-lookup"><span data-stu-id="212aa-116">However, that does not imply that the same entity, with the same attributes and logic would be implemented in multiple Bounded Contexts.</span></span> <span data-ttu-id="212aa-117">Nesse caso, as entidades em cada Contexto Limitado limitam seus atributos e comportamentos aos que são necessários no domínio do Contexto Limitado.</span><span class="sxs-lookup"><span data-stu-id="212aa-117">Instead, entities in each Bounded Context limit their attributes and behaviors to those required in that Bounded Context’s domain.</span></span>

<span data-ttu-id="212aa-118">Por exemplo, a entidade de comprador talvez tenha a maioria dos atributos de uma pessoa que estão definidos na entidade de usuário no microsserviço de perfil ou de identidade, incluindo a identidade.</span><span class="sxs-lookup"><span data-stu-id="212aa-118">For instance, the buyer entity might have most of a person’s attributes that are defined in the user entity in the profile or identity microservice, including the identity.</span></span> <span data-ttu-id="212aa-119">Mas a entidade de comprador no microsserviço de pedidos pode ter menos atributos, porque somente determinados dados do comprador estão relacionados ao processo de pedido.</span><span class="sxs-lookup"><span data-stu-id="212aa-119">But the buyer entity in the ordering microservice might have fewer attributes, because only certain buyer data is related to the order process.</span></span> <span data-ttu-id="212aa-120">O contexto de cada microsserviço ou o Contexto Limitado afeta o modelo de domínio.</span><span class="sxs-lookup"><span data-stu-id="212aa-120">The context of each microservice or Bounded Context impacts its domain model.</span></span>

<span data-ttu-id="212aa-121">*As entidades de domínio precisam implementar o comportamento, além de implementar os atributos de dados*</span><span class="sxs-lookup"><span data-stu-id="212aa-121">*Domain entities must implement behavior in addition to implementing data attributes*</span></span>

<span data-ttu-id="212aa-122">Uma entidade de domínio no DDD precisa implementar a lógica do domínio ou o comportamento relacionado aos dados da entidade (o objeto acessado na memória).</span><span class="sxs-lookup"><span data-stu-id="212aa-122">A domain entity in DDD must implement the domain logic or behavior related to the entity data (the object accessed in memory).</span></span> <span data-ttu-id="212aa-123">Por exemplo, como parte de uma classe de entidade de pedido, você precisa ter a lógica de negócios e as operações implementadas como métodos para tarefas como adicionar um item de pedido, validação de dados e cálculo de total.</span><span class="sxs-lookup"><span data-stu-id="212aa-123">For example, as part of an order entity class you must have business logic and operations implemented as methods for tasks such as adding an order item, data validation, and total calculation.</span></span> <span data-ttu-id="212aa-124">Os métodos da entidade encarregam-se das invariáveis e das regras da entidade em vez de disseminar essas regras na camada de aplicativo.</span><span class="sxs-lookup"><span data-stu-id="212aa-124">The entity’s methods take care of the invariants and rules of the entity instead of having those rules spread across the application layer.</span></span>

### <a name="rich-domain-model-versus-anemic-domain-model"></a><span data-ttu-id="212aa-131">Modelo de domínio avançado em comparação com o modelo de domínio anêmico</span><span class="sxs-lookup"><span data-stu-id="212aa-131">Rich domain model versus anemic domain model</span></span>

<span data-ttu-id="212aa-132">Em sua postagem [AnemicDomainModel](https://martinfowler.com/bliki/AnemicDomainModel.html), Martin Fowler descreve um modelo de domínio fraco desta forma:</span><span class="sxs-lookup"><span data-stu-id="212aa-132">In his post [AnemicDomainModel](https://martinfowler.com/bliki/AnemicDomainModel.html), Martin Fowler describes an anemic domain model this way:</span></span>

<span data-ttu-id="212aa-133">O sintoma básico de um modelo de domínio anêmico é que, à primeira vista, ele parece real.</span><span class="sxs-lookup"><span data-stu-id="212aa-133">The basic symptom of an Anemic Domain Model is that at first blush it looks like the real thing.</span></span> <span data-ttu-id="212aa-134">Existem objetos, muitos nomeados de acordo com os nomes no espaço de domínio, conectados com as relações avançadas e a estrutura que os modelos de domínio verdadeiros têm.</span><span class="sxs-lookup"><span data-stu-id="212aa-134">There are objects, many named after the nouns in the domain space, and these objects are connected with the rich relationships and structure that true domain models have.</span></span> <span data-ttu-id="212aa-135">A verdade aparece quando você observa o comportamento e percebe que não há praticamente nenhum comportamento nesses objetos, ou seja, eles não passam de pacotes de getters e setters.</span><span class="sxs-lookup"><span data-stu-id="212aa-135">The catch comes when you look at the behavior, and you realize that there is hardly any behavior on these objects, making them little more than bags of getters and setters.</span></span>

<span data-ttu-id="212aa-136">É claro que, quando você usar um modelo de domínio anêmico, serão usados os modelos de dados de um conjunto de objetos de serviço (tradicionalmente chamado de *camada comercial*) que captura toda a lógica de negócios ou do domínio.</span><span class="sxs-lookup"><span data-stu-id="212aa-136">Of course, when you use an anemic domain model, those data models will be used from a set of service objects (traditionally named the *business layer*) which capture all the domain or business logic.</span></span> <span data-ttu-id="212aa-137">A camada de negócios fica acima do modelo de dados e usa o modelo de dados apenas como dados.</span><span class="sxs-lookup"><span data-stu-id="212aa-137">The business layer sits on top of the data model and uses the data model just as data.</span></span>

<span data-ttu-id="212aa-138">O modelo de domínio anêmico é apenas um design de estilo de procedimento.</span><span class="sxs-lookup"><span data-stu-id="212aa-138">The anemic domain model is just a procedural style design.</span></span> <span data-ttu-id="212aa-139">Os objetos da entidade anêmica não são objetos reais, porque eles não têm comportamento (métodos).</span><span class="sxs-lookup"><span data-stu-id="212aa-139">Anemic entity objects are not real objects because they lack behavior (methods).</span></span> <span data-ttu-id="212aa-140">Eles apenas mantêm as propriedades dos dados e, portanto, não são de design orientado a objeto.</span><span class="sxs-lookup"><span data-stu-id="212aa-140">They only hold data properties and thus it is not object-oriented design.</span></span> <span data-ttu-id="212aa-141">Ao colocar todo o comportamento nos objetos de serviço (na camada de negócios), basicamente, o resultado será um [código espaguete](https://en.wikipedia.org/wiki/Spaghetti_code) ou [scripts de transação](https://martinfowler.com/eaaCatalog/transactionScript.html) e, portanto, você perderá as vantagens que um modelo de domínio oferece.</span><span class="sxs-lookup"><span data-stu-id="212aa-141">By putting all the behavior out into service objects (the business layer) you essentially end up with [spaghetti code](https://en.wikipedia.org/wiki/Spaghetti_code) or [transaction scripts](https://martinfowler.com/eaaCatalog/transactionScript.html), and therefore you lose the advantages that a domain model provides.</span></span>

<span data-ttu-id="212aa-142">De qualquer maneira, se o microsserviço ou o Contexto Limitado for muito simples (um serviço de CRUD), o modelo de domínio anêmico na forma de objetos de entidade apenas com as propriedades dos dados já poderá ser suficiente e talvez não compense implementar os padrões mais complexos de DDD.</span><span class="sxs-lookup"><span data-stu-id="212aa-142">Regardless, if your microservice or Bounded Context is very simple (a CRUD service), the anemic domain model in the form of entity objects with just data properties might be good enough, and it might not be worth implementing more complex DDD patterns.</span></span> <span data-ttu-id="212aa-143">Nesse caso, ele será simplesmente um modelo de persistência, porque você criou uma entidade intencionalmente apenas com os dados para fins de CRUD.</span><span class="sxs-lookup"><span data-stu-id="212aa-143">In that case, it will be simply a persistence model, because you have intentionally created an entity with only data for CRUD purposes.</span></span>

<span data-ttu-id="212aa-144">É por isso que as arquiteturas de microsserviços são perfeitas para uma abordagem de várias arquiteturas, dependendo de cada Contexto Limitado.</span><span class="sxs-lookup"><span data-stu-id="212aa-144">That is why microservices architectures are perfect for a multi-architectural approach depending on each Bounded Context.</span></span> <span data-ttu-id="212aa-145">Por exemplo, no eShopOnContainers, o microsserviço de pedidos implementa os padrões de DDD, mas o microsserviço de catálogo, que é um serviço de CRUD simples, não implementa.</span><span class="sxs-lookup"><span data-stu-id="212aa-145">For instance, in eShopOnContainers, the ordering microservice implements DDD patterns, but the catalog microservice, which is a simple CRUD service, does not.</span></span>

<span data-ttu-id="212aa-146">Algumas pessoas dizem que o modelo de domínio anêmico é um antipadrão.</span><span class="sxs-lookup"><span data-stu-id="212aa-146">Some people say that the anemic domain model is an anti-pattern.</span></span> <span data-ttu-id="212aa-147">Isso realmente depende do que está sendo implementando.</span><span class="sxs-lookup"><span data-stu-id="212aa-147">It really depends on what you are implementing.</span></span> <span data-ttu-id="212aa-148">Se o microsserviço que você estiver criando for simples o suficiente (por exemplo, um serviço de CRUD), seguir o modelo de domínio anêmico não será um antipadrão.</span><span class="sxs-lookup"><span data-stu-id="212aa-148">If the microservice you are creating is simple enough (for example, a CRUD service), following the anemic domain model it is not an anti-pattern.</span></span> <span data-ttu-id="212aa-149">No entanto, se você precisar lidar com a complexidade de um domínio de microsserviço que tenha várias regras de negócios em constante mudança, o modelo de domínio anêmico poderá ser um antipadrão para esse microsserviço ou Contexto Limitado.</span><span class="sxs-lookup"><span data-stu-id="212aa-149">However, if you need to tackle the complexity of a microservice’s domain that has a lot of ever-changing business rules, the anemic domain model might be an anti-pattern for that microservice or Bounded Context.</span></span> <span data-ttu-id="212aa-150">Nesse caso, criá-lo como um modelo avançado com entidades contendo dados e comportamentos, bem como implementar os padrões de DDD adicionais (agregações, objetos de valor, etc.) pode trazer enormes benefícios para o sucesso a longo prazo desse microsserviço.</span><span class="sxs-lookup"><span data-stu-id="212aa-150">In that case, designing it as a rich model with entities containing data plus behavior as well as implementing additional DDD patterns (aggregates, value objects, etc.) might have huge benefits for the long-term success of such a microservice.</span></span>

### <a name="the-value-object-pattern"></a><span data-ttu-id="212aa-156">O padrão de objeto de valor</span><span class="sxs-lookup"><span data-stu-id="212aa-156">The Value Object pattern</span></span>

<span data-ttu-id="212aa-157">Conforme observado por Eric Evans, "Muitos objetos não têm uma identidade conceitual.</span><span class="sxs-lookup"><span data-stu-id="212aa-157">As Eric Evans has noted, “Many objects do not have conceptual identity.</span></span> <span data-ttu-id="212aa-158">Esses objetos descrevem certas características de algo.”</span><span class="sxs-lookup"><span data-stu-id="212aa-158">These objects describe certain characteristics of a thing.”</span></span>

<span data-ttu-id="212aa-159">Uma entidade requer uma identidade, mas há muitos objetos em um sistema que não requerem, como o padrão de objeto de valor.</span><span class="sxs-lookup"><span data-stu-id="212aa-159">An entity requires an identity, but there are many objects in a system that do not, like the Value Object pattern.</span></span> <span data-ttu-id="212aa-160">Um objeto de valor é um objeto sem nenhuma identidade conceitual que descreve um aspecto de domínio.</span><span class="sxs-lookup"><span data-stu-id="212aa-160">A value object is an object with no conceptual identity that describes a domain aspect.</span></span> <span data-ttu-id="212aa-161">Esses são objetos dos quais uma instância é criada para representar os elementos de design que interessam apenas temporariamente.</span><span class="sxs-lookup"><span data-stu-id="212aa-161">These are objects that you instantiate to represent design elements that only concern you temporarily.</span></span> <span data-ttu-id="212aa-162">Importa *o que* eles são e não *quem* são.</span><span class="sxs-lookup"><span data-stu-id="212aa-162">You care about *what* they are, not *who* they are.</span></span> <span data-ttu-id="212aa-163">Exemplos incluem números e cadeias de caracteres, mas também podem ser conceitos de nível superior como grupos de atributos.</span><span class="sxs-lookup"><span data-stu-id="212aa-163">Examples include numbers and strings, but can also be higher-level concepts like groups of attributes.</span></span>

<span data-ttu-id="212aa-164">Algo que é uma entidade em um microsserviço pode não ser uma entidade em outro microsserviço, porque no segundo caso, o Contexto Limitado pode ter um significado <span data-ttu-id="212aa-170">Uma pessoa com um nome e um sobrenome geralmente é uma entidade, porque a pessoa tem uma identidade, mesmo se o nome e o sobrenome coincidirem com outro conjunto de valores, por exemplo, se esses nomes também se referirem a uma outra pessoa.</span><span class="sxs-lookup"><span data-stu-id="212aa-170">A person with a name and surname is usually an entity because a person has identity, even if the name and surname coincide with another set of values, such as if those names also refers to a different person.</span></span>

<span data-ttu-id="212aa-171">Os objetos de valor são difíceis de gerenciar em bancos de dados relacionais e em ORMs, como o EF, mas em bancos de dados orientados a documentos eles são mais fáceis de implementar e usar.</span><span class="sxs-lookup"><span data-stu-id="212aa-171">Value objects are hard to manage in relational databases and ORMs like EF, whereas in document oriented databases they are easier to implement and use.</span></span>

### <a name="the-aggregate-pattern"></a><span data-ttu-id="212aa-178">O padrão de agregação</span><span class="sxs-lookup"><span data-stu-id="212aa-178">The Aggregate pattern</span></span>

<span data-ttu-id="212aa-179">Um modelo de domínio contém diferentes clusters de entidades de dados e processos que podem controlar uma área significativa de funcionalidades, como inventário ou execução de pedidos.</span><span class="sxs-lookup"><span data-stu-id="212aa-179">A domain model contains clusters of different data entities and processes that can control a significant area of functionality, such as order fulfilment or inventory.</span></span> <span data-ttu-id="212aa-180">Uma unidade de DDD mais refinada é a agregação, que descreve um cluster ou o grupo de entidades e os comportamentos que podem ser tratados como uma unidade coesa.</span><span class="sxs-lookup"><span data-stu-id="212aa-180">A more fine-grained DDD unit is the aggregate, which describes a cluster or group of entities and behaviors that can be treated as a cohesive unit.</span></span>

<span data-ttu-id="212aa-181">Normalmente, a agregação é definida com base nas transações que são necessárias.</span><span class="sxs-lookup"><span data-stu-id="212aa-181">You usually define an aggregate based on the transactions that you need.</span></span> <span data-ttu-id="212aa-182">Um exemplo clássico é um pedido que também contém uma lista de itens do pedido.</span><span class="sxs-lookup"><span data-stu-id="212aa-182">A classic example is an order that also contains a list of order items.</span></span> <span data-ttu-id="212aa-183">Geralmente, um item do pedido será uma entidade.</span><span class="sxs-lookup"><span data-stu-id="212aa-183">An order item will usually be an entity.</span></span> <span data-ttu-id="212aa-184">Mas ela será uma entidade filha dentro da agregação de pedido, que também conterá a entidade de pedido como entidade raiz, geralmente chamada de raiz de agregação.</span><span class="sxs-lookup"><span data-stu-id="212aa-184">But it will be a child entity within the order aggregate, which will also contain the order entity as its root entity, typically called an aggregate root.</span></span>

<span data-ttu-id="212aa-185">Pode ser difícil identificar agregações.</span><span class="sxs-lookup"><span data-stu-id="212aa-185">Identifying aggregates can be hard.</span></span> <span data-ttu-id="212aa-186">Uma agregação é um grupo de objetos que precisam ser consistentes em conjunto, mas não basta apenas selecionar um grupo de objetos e rotulá-lo como uma agregação.</span><span class="sxs-lookup"><span data-stu-id="212aa-186">An aggregate is a group of objects that must be consistent together, but you cannot just pick a group of objects and label them an aggregate.</span></span> <span data-ttu-id="212aa-187">Você precisa começar com um conceito de domínio e pensar sobre as entidades que são usadas nas transações mais comuns relacionadas a esse conceito.</span><span class="sxs-lookup"><span data-stu-id="212aa-187">You must start with a domain concept and think about the entities that are used in the most common transactions related to that concept.</span></span> <span data-ttu-id="212aa-188">As entidades que precisam ser consistentes entre as transações são as que formam uma agregação.</span><span class="sxs-lookup"><span data-stu-id="212aa-188">Those entities that need to be transactionally consistent are what forms an aggregate.</span></span> <span data-ttu-id="212aa-189">Pensar sobre as operações de transação provavelmente é a melhor maneira de identificar as agregações.</span><span class="sxs-lookup"><span data-stu-id="212aa-189">Thinking about transaction operations is probably the best way to identify aggregates.</span></span>

### <a name="the-aggregate-root-or-root-entity-pattern"></a><span data-ttu-id="212aa-190">O padrão de raiz de agregação ou de entidade raiz</span><span class="sxs-lookup"><span data-stu-id="212aa-190">The Aggregate Root or Root Entity pattern</span></span>

<span data-ttu-id="212aa-191">Uma agregação é composta por pelo menos uma entidade: a raiz de agregação, também chamada de entidade raiz ou entidade principal.</span><span class="sxs-lookup"><span data-stu-id="212aa-191">An aggregate is composed of at least one entity: the aggregate root, also called root entity or primary entity.</span></span> <span data-ttu-id="212aa-192">Além disso, ela pode ter várias entidades filhas e objetos de valor, com todas as entidades e os objetos trabalhando juntos para implementar as transações e os comportamentos necessários.</span><span class="sxs-lookup"><span data-stu-id="212aa-192">Additionally, it can have multiple child entities and value objects, with all entities and objects working together to implement required behavior and transactions.</span></span>

<span data-ttu-id="212aa-193">A finalidade de uma raiz de agregação é garantir a consistência da agregação. Ela deve ser o único ponto de entrada para as atualizações da agregação por meio de métodos ou operações na classe raiz de agregação.</span><span class="sxs-lookup"><span data-stu-id="212aa-193">The purpose of an aggregate root is to ensure the consistency of the aggregate; it should be the only entry point for updates to the aggregate through methods or operations in the aggregate root class.</span></span> <span data-ttu-id="212aa-194">Você deve fazer alterações nas entidades na agregação apenas por meio da raiz de agregação.</span><span class="sxs-lookup"><span data-stu-id="212aa-194">You should make changes to entities within the aggregate only via the aggregate root.</span></span> <span data-ttu-id="212aa-195">Ela é a proteção da consistência da agregação, levando em conta todas as invariáveis e as regras de consistência que devem ser obedecidas na agregação.</span><span class="sxs-lookup"><span data-stu-id="212aa-195">It is the aggregate’s consistency guardian, taking into account all the invariants and consistency rules you might need to comply with in your aggregate.</span></span> <span data-ttu-id="212aa-196">Se você alterar uma entidade filha ou um objeto de valor de forma independente, a raiz de agregação não poderá garantir que a agregação esteja em um estado válido.</span><span class="sxs-lookup"><span data-stu-id="212aa-196">If you change a child entity or value object independently, the aggregate root cannot ensure that the aggregate is in a valid state.</span></span> <span data-ttu-id="212aa-197">Isso seria semelhante a uma tabela com um segmento flexível.</span><span class="sxs-lookup"><span data-stu-id="212aa-197">It would be like a table with a loose leg.</span></span> <span data-ttu-id="212aa-198">Manter a consistência é o principal objetivo da raiz de agregação.</span><span class="sxs-lookup"><span data-stu-id="212aa-198">Maintaining consistency is the main purpose of the aggregate root.</span></span>

<span data-ttu-id="212aa-199">Na Figura 9-9, veja as agregações de exemplo, como a agregação de comprador, que contém uma única entidade (a raiz de agregação de comprador).</span><span class="sxs-lookup"><span data-stu-id="212aa-199">In Figure 9-9, you can see sample aggregates like the buyer aggregate, which contains a single entity (the aggregate root Buyer).</span></span> <span data-ttu-id="212aa-200">A agregação de pedido contém várias entidades e um objeto de valor.</span><span class="sxs-lookup"><span data-stu-id="212aa-200">The order aggregate contains multiple entities and a value object.</span></span>
