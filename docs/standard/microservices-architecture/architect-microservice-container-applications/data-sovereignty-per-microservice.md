# <a name="data-sovereignty-per-microservice"></a><span data-ttu-id="f0513-104">Soberania dados por microsserviço</span><span class="sxs-lookup"><span data-stu-id="f0513-104">Data sovereignty per microservice</span></span>

<span data-ttu-id="f0513-105">Uma regra importante para arquitetura microservices é que cada microsserviço deve ter seus dados de domínio e lógica.</span><span class="sxs-lookup"><span data-stu-id="f0513-105">An important rule for microservices architecture is that each microservice must own its domain data and logic.</span></span> <span data-ttu-id="f0513-106">Assim como um aplicativo completo possui sua lógica e os dados, portanto deve cada microsserviço possui sua lógica e os dados em um ciclo de vida autônomo, com a implantação independente por microsserviço.</span><span class="sxs-lookup"><span data-stu-id="f0513-106">Just as a full application owns its logic and data, so must each microservice own its logic and data under an autonomous lifecycle, with independent deployment per microservice.</span></span>

<span data-ttu-id="f0513-107">Isso significa que o modelo conceitual do domínio será diferente entre subsistemas ou microservices.</span><span class="sxs-lookup"><span data-stu-id="f0513-107">This means that the conceptual model of the domain will differ between subsystems or microservices.</span></span> <span data-ttu-id="f0513-108">Considere a possibilidade de aplicativos da empresa, onde os aplicativos de CRM (gerenciamento) relação cliente, transacionais comprarem subsistemas e subsistemas de suporte ao cliente cada chamada em atributos de entidade exclusivo do cliente e dados, e cada emprega um diferente Contexto limitado (BC).</span><span class="sxs-lookup"><span data-stu-id="f0513-108">Consider enterprise applications, where customer relationship management (CRM) applications, transactional purchase subsystems, and customer support subsystems each call on unique customer entity attributes and data, and where each employs a different Bounded Context (BC).</span></span>

<span data-ttu-id="f0513-109">Esse princípio é semelhante em [design orientado a domínio (DDD)](https://en.wikipedia.org/wiki/Domain-driven_design), onde cada [limitada contexto](https://martinfowler.com/bliki/BoundedContext.html) ou subsistema autônomo ou serviço deve possuir o seu modelo de domínio (dados mais lógica e comportamento).</span><span class="sxs-lookup"><span data-stu-id="f0513-109">This principle is similar in [Domain-driven design (DDD)](https://en.wikipedia.org/wiki/Domain-driven_design), where each [Bounded Context](https://martinfowler.com/bliki/BoundedContext.html) or autonomous subsystem or service must own its domain model (data plus logic and behavior).</span></span> <span data-ttu-id="f0513-110">Cada DDD limitada contexto se correlaciona com uma microsserviço de negócios (um ou vários serviços).</span><span class="sxs-lookup"><span data-stu-id="f0513-110">Each DDD Bounded Context correlates to one business microservice (one or several services).</span></span> <span data-ttu-id="f0513-111">(É expandir nesse ponto sobre o padrão do contexto associado na próxima seção).</span><span class="sxs-lookup"><span data-stu-id="f0513-111">(We expand on this point about the Bounded Context pattern in the next section.)</span></span>

<span data-ttu-id="f0513-112">Por outro lado, a abordagem tradicional (dados monolítico) usada em muitos aplicativos é ter um único banco de dados centralizado ou apenas alguns bancos de dados.</span><span class="sxs-lookup"><span data-stu-id="f0513-112">On the other hand, the traditional (monolithic data) approach used in many applications is to have a single centralized database or just a few databases.</span></span> <span data-ttu-id="f0513-113">Isso geralmente é um banco de dados SQL normalizado que é usado para o aplicativo inteiro e todos os seus subsistemas internos, conforme mostrado na Figura 4-7.</span><span class="sxs-lookup"><span data-stu-id="f0513-113">This is often a normalized SQL database that is used for the whole application and all its internal subsystems, as shown in Figure 4-7.</span></span>

<span data-ttu-id="f0513-116">Inicialmente, a abordagem de banco de dados centralizado é mais simples e parece habilitar a reutilização de entidades em subsistemas diferentes para fazer tudo o que é consistente.</span><span class="sxs-lookup"><span data-stu-id="f0513-116">The centralized database approach initially looks simpler and seems to enable reuse of entities in different subsystems to make everything consistent.</span></span> <span data-ttu-id="f0513-117">Mas a realidade é que você acabará com grandes tabelas que servem vários subsistemas diferentes e que incluem atributos e colunas que não são necessários na maioria dos casos.</span><span class="sxs-lookup"><span data-stu-id="f0513-117">But the reality is you end up with huge tables that serve many different subsystems, and that include attributes and columns that are not needed in most cases.</span></span> <span data-ttu-id="f0513-118">é como tentar usar o mesmo mapa físico para caminhada uma trilha de curta, fazer uma viagem de carro de dia inteiro e aprendizado Geografia.</span><span class="sxs-lookup"><span data-stu-id="f0513-118">it is like trying to use the same physical map for hiking a short trail, taking a day-long car trip, and learning geography.</span></span>

<span data-ttu-id="f0513-119">Um aplicativo monolítico com normalmente um único banco de dados relacional tem dois benefícios importantes: [transações ACID](https://en.wikipedia.org/wiki/ACID) e a linguagem SQL, ambos os funcionando em todas as tabelas e os dados relacionados a seu aplicativo.</span><span class="sxs-lookup"><span data-stu-id="f0513-119">A monolithic application with typically a single relational database has two important benefits: [ACID transactions](https://en.wikipedia.org/wiki/ACID) and the SQL language, both working across all the tables and data related to your application.</span></span> <span data-ttu-id="f0513-120">Essa abordagem fornece uma maneira fácil escrever uma consulta que combina dados de várias tabelas.</span><span class="sxs-lookup"><span data-stu-id="f0513-120">This approach provides a way to easily write a query that combines data from multiple tables.</span></span>

microsserviço.</span><span class="sxs-lookup"><span data-stu-id="f0513-122">But even when ACID transactions can or should be used within a microservice or Bounded Context, the data owned by each <span data-ttu-id="f0513-127">Por sua vez, isso significa que você deve usar consistência eventual quando um processo de negócios abrange vários microservices.</span><span class="sxs-lookup"><span <span data-ttu-id="f0513-129">Indo mais além, microservices diferentes geralmente usam diferentes *tipos* de bancos de dados.</span><span class="sxs-lookup"><span data-stu-id="f0513-129">Going even ## <a name="the-relationship-between-microservices-and-the-bounded-context-pattern"></a><span data-ttu-id="f0513-137">A relação entre microservices e o padrão de contexto limitado</span><span class="sxs-lookup"><span data-stu-id="f0513-137">The relationship between microservices and the Bounded Context pattern</span></span>

<span data-ttu-id="f0513-138">O conceito de microsserviço deriva o [padrão do contexto associado (BC)](http://martinfowler.com/bliki/BoundedContext.html) na [design orientado a domínio (DDD)](https://en.wikipedia.org/wiki/Domain-driven_design).</span><span class="sxs-lookup"><span data-stu-id="f0513-138">The concept of microservice derives from the [Bounded Context (BC) pattern](http://martinfowler.com/bliki/BoundedContext.html) in [domain-driven design (DDD)](https://en.wikipedia.org/wiki/Domain-driven_design).</span></span> <span data-ttu-id="f0513-139">DDD lida com modelos grandes, dividindo-los em vários BCs e sendo explícito sobre seus limites.</span><span class="sxs-lookup"><span data-stu-id="f0513-139">DDD deals with large models by dividing them into multiple BCs and being explicit about their boundaries.</span></span> <span data-ttu-id="f0513-140">Cada BC deve ter seu próprio modelo e o banco de dados. da mesma forma, cada microsserviço possui seus dados relacionados.</span><span class="sxs-lookup"><span data-stu-id="f0513-140">Each BC must have its own model and database; likewise, each microservice owns its related data.</span></span> <span data-ttu-id="f0513-141">Além disso, cada BC geralmente tem seu próprio [idioma onipresente](http://martinfowler.com/bliki/UbiquitousLanguage.html) para ajudar a comunicação entre os desenvolvedores de software e especialistas de domínio.</span><span class="sxs-lookup"><span data-stu-id="f0513-141">In addition, each BC usually has its own [ubiquitous language](http://martinfowler.com/bliki/UbiquitousLanguage.html) to help communication between software developers and domain experts.</span></span>

<span data-ttu-id="f0513-142">Esses termos (principalmente entidades de domínio) no idioma onipresente podem ter nomes diferentes em diferentes contextos limitada, mesmo quando outro domínio entidades compartilham a mesma identidade (ou seja, a ID exclusiva que é usada para ler a entidade de armazenamento).</span><span class="sxs-lookup"><span data-stu-id="f0513-142">Those terms (mainly domain entities) in the ubiquitous language can have different names in different Bounded Contexts, even when different domain entities share the same identity (that is, the unique ID that is used to read the entity from storage).</span></span> <span data-ttu-id="f0513-143">Por exemplo, em um perfil de usuário limitado contexto, a entidade de domínio do usuário pode compartilhar identidade com a entidade de domínio do comprador no contexto de ordenação limitada.</span><span class="sxs-lookup"><span data-stu-id="f0513-143">For instance, in a user-profile Bounded Context, the User domain entity might share identity with the Buyer domain entity in the ordering Bounded Context.</span></span>

<span data-ttu-id="f0513-144">Um microsserviço, portanto, é como um contexto limitado, mas ela também especifica que se trata de um serviço distribuído.</span><span class="sxs-lookup"><span data-stu-id="f0513-144">A microservice is therefore like a Bounded Context, but it also specifies that it is a distributed service.</span></span> <span data-ttu-id="f0513-145">Ele é criado como um processo separado para cada contexto associado e deve usar os protocolos distribuídos observados anteriormente, como HTTP/HTTPS, o WebSocket, ou [AMQP](https://en.wikipedia.org/wiki/Advanced_Message_Queuing_Protocol).</span><span class="sxs-lookup"><span data-stu-id="f0513-145">It is built as a separate process for each Bounded Context, and it must use the distributed protocols noted earlier, like HTTP/HTTPS, WebSockets, or [AMQP](https://en.wikipedia.org/wiki/Advanced_Message_Queuing_Protocol).</span></span> <span data-ttu-id="f0513-146">O padrão de contexto limitado, no entanto, não especifica se o contexto associado é um serviço distribuído ou se ele é simplesmente um limite lógico (por exemplo, um subsistema genérico) dentro de um aplicativo de implantação monolítico.</span><span class="sxs-lookup"><span data-stu-id="f0513-146">The Bounded Context pattern, however, does not specify whether the Bounded Context is a distributed service or if it is simply a logical boundary (such as a generic subsystem) within a monolithic-deployment application.</span></span>

<span data-ttu-id="f0513-147">É importante destacar que a definição de um serviço para cada contexto associado é um bom lugar para começar.</span><span class="sxs-lookup"><span data-stu-id="f0513-147">It is important to highlight that defining a service for each Bounded Context is a good place to start.</span></span> <span data-ttu-id="f0513-148">Mas você não precisa restringir o design para ele.</span><span class="sxs-lookup"><span data-stu-id="f0513-148">But you do not have to constrain your design to it.</span></span> <span data-ttu-id="f0513-149">Às vezes, você deve criar um contexto associado ou business microsserviço composto por vários serviços físicos.</span><span class="sxs-lookup"><span data-stu-id="f0513-149">Sometimes you must design a Bounded Context or business microservice composed of several physical services.</span></span> <span data-ttu-id="f0513-150">Mas, por fim, ambos os padrões — contexto limitado e microsserviço — estão intimamente relacionados.</span><span class="sxs-lookup"><span data-stu-id="f0513-150">But ultimately, both patterns—Bounded Context and microservice—are closely related.</span></span>

<span data-ttu-id="f0513-151">DDD beneficia microservices obtendo os limites reais na forma de microservices distribuída.</span><span class="sxs-lookup"><span data-stu-id="f0513-151">DDD benefits from microservices by getting real boundaries in the form of distributed microservices.</span></span> <span data-ttu-id="f0513-152">Mas ideias, como o modelo entre microservices de compartilhamento não são o que você deseja em um contexto associado.</span><span class="sxs-lookup"><span data-stu-id="f0513-152">But ideas like not sharing the model between microservices are what you also want in a Bounded Context.</span></span>
