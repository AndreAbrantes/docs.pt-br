## <a name="challenge-4-how-to-design-communication-across-microservice-boundaries"></a><span data-ttu-id="ac627-178">Desafio \#4: como projetar a comunicação entre os limites dos microsserviços</span><span class="sxs-lookup"><span data-stu-id="ac627-178">Challenge \#4: How to design communication across microservice boundaries</span></span>

<span data-ttu-id="ac627-179">A comunicação entre os limites dos microsserviços é realmente um grande desafio.</span><span class="sxs-lookup"><span data-stu-id="ac627-179">Communicating across microservice boundaries is a real challenge.</span></span> <span data-ttu-id="ac627-180">Nesse contexto, a comunicação não se refere a qual protocolo você deve usar (HTTP e REST, AMQP, mensagens e assim por diante).</span><span class="sxs-lookup"><span data-stu-id="ac627-180">In this context, communication doesn't refer to what protocol you should use (HTTP and REST, AMQP, messaging, and so on).</span></span> <span data-ttu-id="ac627-181">Nesse caso, ela aborda qual estilo de comunicação você deve usar e, principalmente, que nível de acoplamento os microsserviços devem ter.</span><span class="sxs-lookup"><span data-stu-id="ac627-181">Instead, it addresses what communication style you should use, and especially how coupled your microservices should be.</span></span> <span data-ttu-id="ac627-182">Dependendo do nível de acoplamento, quando ocorrer uma falha, o impacto dessa falha no sistema poderá variar significativamente.</span><span class="sxs-lookup"><span data-stu-id="ac627-182">Depending on the level of coupling, when failure occurs, the impact of that failure on your system will vary significantly.</span></span>

<span data-ttu-id="ac627-183">Em um sistema distribuído, como um aplicativo baseado em microsserviços, com um número muito grande de artefatos em movimentação, com serviços distribuídos em vários servidores ou hosts, eventualmente, os componentes falharão.</span><span class="sxs-lookup"><span data-stu-id="ac627-183">In a distributed system like a microservices-based application, with so many artifacts moving around and with distributed services across many servers or hosts, components will eventually fail.</span></span> <span data-ttu-id="ac627-184">Falhas parciais e interrupções ainda maiores certamente ocorrerão, portanto, você precisa projetar seus microsserviços e a comunicação entre eles levando em conta os riscos comuns nesse tipo de sistema distribuído.</span><span class="sxs-lookup"><span data-stu-id="ac627-184">Partial failure and even larger outages will occur, so you need to design your microservices and the communication across them considering the common risks in this type of distributed system.</span></span>

<span data-ttu-id="ac627-185">Uma abordagem comum é implementar microsserviços baseados em HTTP (REST), devido à simplicidade.</span><span class="sxs-lookup"><span data-stu-id="ac627-185">A popular approach is to implement HTTP (REST)-based microservices, due to their simplicity.</span></span> <span data-ttu-id="ac627-186">Uma abordagem baseada em HTTP é perfeitamente aceitável e, nesse caso, o problema está relacionado à maneira de usá-la.</span><span class="sxs-lookup"><span data-stu-id="ac627-186">An HTTP-based approach is perfectly acceptable; the issue here is related to how you use it.</span></span> <span data-ttu-id="ac627-187">Se você usar solicitações e respostas HTTP apenas para que os aplicativos cliente ou os gateways de API interajam com os microsserviços, tudo bem.</span><span class="sxs-lookup"><span data-stu-id="ac627-187">If you use HTTP requests and responses just to interact with your microservices from client applications or from API Gateways, that's fine.</span></span> <span data-ttu-id="ac627-188">Mas, se você criar longas cadeias de chamadas HTTP síncronas entre os microsserviços, comunicando-se entre seus limites como se os microsserviços fossem objetos em um aplicativo monolítico, seu aplicativo acabará tendo problemas.</span><span class="sxs-lookup"><span data-stu-id="ac627-188">But if you create long chains of synchronous HTTP calls across microservices, communicating across their boundaries as if the microservices were objects in a monolithic application, your application will eventually run into problems.</span></span>

<span data-ttu-id="ac627-189">Por exemplo, imagine que o aplicativo cliente faça uma chamada HTTP à API para um microsserviço individual, como o microsserviço de pedidos.</span><span class="sxs-lookup"><span data-stu-id="ac627-189">For instance, imagine that your client application makes an HTTP API call to an individual microservice like the Ordering microservice.</span></span> <span data-ttu-id="ac627-190">Se o microsserviço de pedidos, por sua vez, chamar microsserviços adicionais usando HTTP no mesmo ciclo de solicitação/resposta, você estará criando uma cadeia de chamadas HTTP.</span><span class="sxs-lookup"><span data-stu-id="ac627-190">If the Ordering microservice in turn calls additional microservices using HTTP within the same request/response cycle, you're creating a chain of HTTP calls.</span></span> <span data-ttu-id="ac627-191">Parece razoável em um primeiro momento.</span><span class="sxs-lookup"><span data-stu-id="ac627-191">It might sound reasonable initially.</span></span> <span data-ttu-id="ac627-192">No entanto, existem pontos importantes a serem considerados ao adotar esse caminho:</span><span class="sxs-lookup"><span data-stu-id="ac627-192">However, there are important points to consider when going down this path:</span></span>

- <span data-ttu-id="ac627-193">Bloqueio e baixo desempenho.</span><span class="sxs-lookup"><span data-stu-id="ac627-193">Blocking and low performance.</span></span> <span data-ttu-id="ac627-194">Devido à natureza síncrona de HTTP, a solicitação original não obtém uma resposta até que todas as chamadas HTTP internas sejam concluídas.</span><span class="sxs-lookup"><span data-stu-id="ac627-194">Due to the synchronous nature of HTTP, the original request doesn't get a response until all the internal HTTP calls are finished.</span></span> <span data-ttu-id="ac627-195">Imagine se o número dessas chamadas aumentar significativamente e ao mesmo tempo uma das chamadas HTTP intermediárias para um microsserviço for bloqueada.</span><span class="sxs-lookup"><span data-stu-id="ac627-195">Imagine if the number of these calls increases significantly and at the same time one of the intermediate HTTP calls to a microservice is blocked.</span></span> <span data-ttu-id="ac627-196">O resultado é que o desempenho será afetado e a escalabilidade geral será extremamente afetada com o aumento de solicitações HTTP adicionais.</span><span class="sxs-lookup"><span data-stu-id="ac627-196">The result is that performance is impacted, and the overall scalability will be exponentially affected as additional HTTP requests increase.</span></span>

- <span data-ttu-id="ac627-197">Acoplando microsserviços com HTTP.</span><span class="sxs-lookup"><span data-stu-id="ac627-197">Coupling microservices with HTTP.</span></span> <span data-ttu-id="ac627-198">Os microsserviços empresariais não devem ser acoplados com outros microsserviços empresariais.</span><span class="sxs-lookup"><span data-stu-id="ac627-198">Business microservices shouldn't be coupled with other business microservices.</span></span> <span data-ttu-id="ac627-199">O ideal é que eles não "saibam" da existência de outros microsserviços.</span><span class="sxs-lookup"><span data-stu-id="ac627-199">Ideally, they shouldn't "know" about the existence of other microservices.</span></span> <span data-ttu-id="ac627-200">Se seu aplicativo depender do acoplamento de microsserviços como no exemplo, será praticamente impossível alcançar a autonomia de cada microsserviço.</span><span class="sxs-lookup"><span data-stu-id="ac627-200">If your application relies on coupling microservices as in the example, achieving autonomy per microservice will be almost impossible.</span></span>

- <span data-ttu-id="ac627-201">Falha em um dos microsserviços.</span><span class="sxs-lookup"><span data-stu-id="ac627-201">Failure in any one microservice.</span></span> <span data-ttu-id="ac627-202">Se você implementar uma cadeia de microsserviços vinculados por chamadas HTTP, quando um dos microsserviços falhar (e, com certeza, isso vai ocorrer) toda a cadeia de microsserviços falhará.</span><span class="sxs-lookup"><span data-stu-id="ac627-202">If you implemented a chain of microservices linked by HTTP calls, when any of the microservices fails (and eventually they will fail) the whole chain of microservices will fail.</span></span> <span data-ttu-id="ac627-203">Um sistema baseado em microsserviço deve ser criado para continuar a trabalhar da melhor maneira possível durante falhas parciais.</span><span class="sxs-lookup"><span data-stu-id="ac627-203">A microservice-based system should be designed to continue to work as well as possible during partial failures.</span></span> <span data-ttu-id="ac627-204">Mesmo se você implementar uma lógica do cliente que use novas tentativas com retirada exponencial ou com mecanismos de disjuntor, quanto mais complexas as cadeias de chamadas HTTP forem, mais complexa será a implementação de uma estratégia de falha baseada em HTTP.</span><span class="sxs-lookup"><span data-stu-id="ac627-204">Even if you implement client logic that uses retries with exponential backoff or circuit breaker mechanisms, the more complex the HTTP call chains are, the more complex it is to implement a failure strategy based on HTTP.</span></span>

<span data-ttu-id="ac627-205">Na verdade, se os microsserviços internos estiverem se comunicando por meio da criação de cadeias de solicitações HTTP, conforme descrito, será possível argumentar que você tem um aplicativo monolítico, mas baseado em HTTP entre processos e não em mecanismos de comunicação entre processos.</span><span class="sxs-lookup"><span data-stu-id="ac627-205">In fact, if your internal microservices are communicating by creating chains of HTTP requests as described, it could be argued that you have a monolithic application, but one based on HTTP between processes instead of intra-process communication mechanisms.</span></span>

<span data-ttu-id="ac627-206">Portanto, para impor a autonomia do microsserviço e melhorar a resiliência, você deve minimizar o uso de cadeias de comunicação de solicitação/resposta entre os microsserviços.</span><span class="sxs-lookup"><span data-stu-id="ac627-206">Therefore, in order to enforce microservice autonomy and have better resiliency, you should minimize the use of chains of request/response communication across microservices.</span></span> <span data-ttu-id="ac627-207">É recomendável usar somente a interação assíncrona para a comunicação entre os microsserviços, seja usando a comunicação assíncrona baseada em evento e em mensagem ou usando a sondagem de HTTP (assíncrona), independentemente do ciclo de solicitação/resposta HTTP original.</span><span class="sxs-lookup"><span data-stu-id="ac627-207">It's recommended that you use only asynchronous interaction for inter-microservice communication, either by using asynchronous message- and event-based communication, or by using (asynchronous) HTTP polling independently of the original HTTP request/response cycle.</span></span>

<span data-ttu-id="ac627-208">O uso da comunicação assíncrona será explicado em detalhes mais adiante neste guia nas seções [A integração assíncrona dos microsserviços impõe a autonomia do microsserviço](communication-in-microservice-architecture.md#asynchronous-microservice-integration-enforces-microservices-autonomy) e [Comunicação assíncrona baseada em mensagem](asynchronous-message-based-communication.md).</span><span class="sxs-lookup"><span data-stu-id="ac627-208">The use of asynchronous communication is explained with additional details later in this guide in the sections [Asynchronous microservice integration enforces microservice's autonomy](communication-in-microservice-architecture.md#asynchronous-microservice-integration-enforces-microservices-autonomy) and [Asynchronous message-based communication](asynchronous-message-based-communication.md).</span></span>
