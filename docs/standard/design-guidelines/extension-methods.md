# <a name="extension-methods"></a><span data-ttu-id="f7c1e-102">Métodos de Extensão</span><span class="sxs-lookup"><span data-stu-id="f7c1e-102">Extension Methods</span></span>
<span data-ttu-id="f7c1e-103">Os métodos de extensão são um recurso de linguagem que permite que métodos estáticos sejam chamados usando a sintaxe de chamada de método de instância.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-103">Extension methods are a language feature that allows static methods to be called using instance method call syntax.</span></span> <span data-ttu-id="f7c1e-104">Esses métodos devem ter pelo menos um parâmetro, que representa a instância em que o método deve operar.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-104">These methods must take at least one parameter, which represents the instance the method is to operate on.</span></span>

 <span data-ttu-id="f7c1e-105">A classe que define esses métodos de extensão é conhecida como a classe "patrocinador" e deve ser declarada como estática.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-105">The class that defines such extension methods is referred to as the "sponsor" class, and it must be declared as static.</span></span> <span data-ttu-id="f7c1e-106">Para usar métodos de extensão, é necessário importar o namespace que define a classe patrocinador.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-106">To use extension methods, one must import the namespace defining the sponsor class.</span></span>

 <span data-ttu-id="f7c1e-107">❌ Evite frivolously a definição de métodos de extensão, especialmente em tipos que não são de sua propriedade.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-107">❌ AVOID frivolously defining extension methods, especially on types you don't own.</span></span>

 <span data-ttu-id="f7c1e-108">Se você tiver o código-fonte próprio de um tipo, considere o uso de métodos de instância regular.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-108">If you do own source code of a type, consider using regular instance methods instead.</span></span> <span data-ttu-id="f7c1e-109">Se você não for o proprietário e quiser adicionar um método, tenha muito cuidado.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-109">If you don't own, and you want to add a method, be very careful.</span></span> <span data-ttu-id="f7c1e-110">O uso liberal de métodos de extensão tem o potencial de desobstruir APIs de tipos que não foram projetados para ter esses métodos.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-110">Liberal use of extension methods has the potential of cluttering APIs of types that were not designed to have these methods.</span></span>

 <span data-ttu-id="f7c1e-111">✔️ Considere o uso de métodos de extensão em qualquer um dos seguintes cenários:</span><span class="sxs-lookup"><span data-stu-id="f7c1e-111">✔️ CONSIDER using extension methods in any of the following scenarios:</span></span>

- <span data-ttu-id="f7c1e-112">Para fornecer a funcionalidade auxiliar relevante para cada implementação de uma interface, se dito, a funcionalidade pode ser escrita em termos da interface principal.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-112">To provide helper functionality relevant to every implementation of an interface, if said functionality can be written in terms of the core interface.</span></span> <span data-ttu-id="f7c1e-113">Isso ocorre porque as implementações concretas não podem ser atribuídas de outra forma a interfaces.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-113">This is because concrete implementations cannot otherwise be assigned to interfaces.</span></span> <span data-ttu-id="f7c1e-114">Por exemplo, os `LINQ to Objects` operadores são implementados como métodos de extensão para todos os <xref:System.Collections.Generic.IEnumerable%601> tipos.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-114">For example, the `LINQ to Objects` operators are implemented as extension methods for all <xref:System.Collections.Generic.IEnumerable%601> types.</span></span> <span data-ttu-id="f7c1e-115">Portanto, qualquer `IEnumerable<>` implementação é habilitada automaticamente por LINQ.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-115">Thus, any `IEnumerable<>` implementation is automatically LINQ-enabled.</span></span>

- <span data-ttu-id="f7c1e-116">Quando um método de instância introduzir uma dependência em algum tipo, mas essa dependência quebraria as regras de gerenciamento de dependência.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-116">When an instance method would introduce a dependency on some type, but such a dependency would break dependency management rules.</span></span> <span data-ttu-id="f7c1e-117">Por exemplo, uma dependência de <xref:System.String> to <xref:System.Uri?displayProperty=nameWithType> é provavelmente não desejável e, portanto, `String.ToUri()` o método de instância retornado `System.Uri` seria o design errado de uma perspectiva de gerenciamento de dependência.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-117">For example, a dependency from <xref:System.String> to <xref:System.Uri?displayProperty=nameWithType> is probably not desirable, and so `String.ToUri()` instance method returning `System.Uri` would be the wrong design from a dependency management perspective.</span></span> <span data-ttu-id="f7c1e-118">Um método de extensão estático `Uri.ToUri(this string str)` retornado `System.Uri` seria um design muito melhor.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-118">A static extension method `Uri.ToUri(this string str)` returning `System.Uri` would be a much better design.</span></span>

 <span data-ttu-id="f7c1e-119">❌ Evite definir métodos de extensão no <xref:System.Object?displayProperty=nameWithType> .</span><span class="sxs-lookup"><span data-stu-id="f7c1e-119">❌ AVOID defining extension methods on <xref:System.Object?displayProperty=nameWithType>.</span></span>

 <span data-ttu-id="f7c1e-120">Os usuários do VB não poderão chamar esses métodos em referências de objeto usando a sintaxe do método de extensão.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-120">VB users will not be able to call such methods on object references using the extension method syntax.</span></span> <span data-ttu-id="f7c1e-121">O VB não dá suporte à chamada de tais métodos porque, no VB, a declaração de uma referência como Object força todas as invocações de método nele para serem associadas tardias (o membro real chamado é determinado em tempo de execução), enquanto as associações aos métodos de extensão são determinadas em tempo de compilação (limite inicial).</span><span class="sxs-lookup"><span data-stu-id="f7c1e-121">VB does not support calling such methods because, in VB, declaring a reference as Object forces all method invocations on it to be late bound (actual member called is determined at runtime), while bindings to extension methods are determined at compile-time (early bound).</span></span>

 <span data-ttu-id="f7c1e-122">Observe que a diretriz se aplica a outros idiomas em que o mesmo comportamento de ligação está presente, ou onde os métodos de extensão não têm suporte.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-122">Note that the guideline applies to other languages where the same binding behavior is present, or where extension methods are not supported.</span></span>

 <span data-ttu-id="f7c1e-123">❌ Não coloque os métodos de extensão no mesmo namespace que o tipo estendido, a menos que ele seja para adicionar métodos a interfaces ou para o gerenciamento de dependência.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-123">❌ DO NOT put extension methods in the same namespace as the extended type unless it is for adding methods to interfaces or for dependency management.</span></span>

 <span data-ttu-id="f7c1e-124">❌ Evite definir dois ou mais métodos de extensão com a mesma assinatura, mesmo que eles residam em namespaces diferentes.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-124">❌ AVOID defining two or more extension methods with the same signature, even if they reside in different namespaces.</span></span>

 <span data-ttu-id="f7c1e-125">✔️ Considere definir métodos de extensão no mesmo namespace que o tipo estendido se o tipo é uma interface e se os métodos de extensão devem ser usados na maioria ou em todos os casos.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-125">✔️ CONSIDER defining extension methods in the same namespace as the extended type if the type is an interface and if the extension methods are meant to be used in most or all cases.</span></span>

 <span data-ttu-id="f7c1e-126">❌ Não defina métodos de extensão que implementam um recurso em namespaces normalmente associados a outros recursos.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-126">❌ DO NOT define extension methods implementing a feature in namespaces normally associated with other features.</span></span> <span data-ttu-id="f7c1e-127">Em vez disso, defina-os no namespace associado ao recurso ao qual eles pertencem.</span><span class="sxs-lookup"><span data-stu-id="f7c1e-127">Instead, define them in the namespace associated with the feature they belong to.</span></span>

 <span data-ttu-id="f7c1e-128">❌ Evite a nomenclatura genérica de namespaces dedicados a métodos de extensão (por exemplo, "extensões").</span><span class="sxs-lookup"><span data-stu-id="f7c1e-128">❌ AVOID generic naming of namespaces dedicated to extension methods (e.g., "Extensions").</span></span> <span data-ttu-id="f7c1e-129">Em vez disso, use um nome descritivo (por exemplo, "roteamento").</span><span class="sxs-lookup"><span data-stu-id="f7c1e-129">Use a descriptive name (e.g., "Routing") instead.</span></span>
