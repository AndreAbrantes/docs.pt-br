# <a name="abstractions-abstract-types-and-interfaces"></a><span data-ttu-id="455be-102">Abstrações (tipos e interfaces abstratos)</span><span class="sxs-lookup"><span data-stu-id="455be-102">Abstractions (Abstract Types and Interfaces)</span></span>
<span data-ttu-id="455be-103">Uma abstração é um tipo que descreve um contrato, mas não fornece uma implementação completa do contrato.</span><span class="sxs-lookup"><span data-stu-id="455be-103">An abstraction is a type that describes a contract but does not provide a full implementation of the contract.</span></span> <span data-ttu-id="455be-104">As abstrações geralmente são implementadas como classes ou interfaces abstratas e são fornecidas com um conjunto bem definido de documentação de referência que descreve a semântica necessária dos tipos que implementam o contrato.</span><span class="sxs-lookup"><span data-stu-id="455be-104">Abstractions are usually implemented as abstract classes or interfaces, and they come with a well-defined set of reference documentation describing the required semantics of the types implementing the contract.</span></span> <span data-ttu-id="455be-105">Algumas das abstrações mais importantes no .NET Framework incluem <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>e <xref:System.Object>.</span><span class="sxs-lookup"><span data-stu-id="455be-105">Some of the most important abstractions in the .NET Framework include <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>, and <xref:System.Object>.</span></span>

 <span data-ttu-id="455be-106">Você pode estender estruturas implementando um tipo concreto que dá suporte ao contrato de uma abstração e usando esse tipo concreto com APIs de estrutura consumindo (operando em) a abstração.</span><span class="sxs-lookup"><span data-stu-id="455be-106">You can extend frameworks by implementing a concrete type that supports the contract of an abstraction and using this concrete type with framework APIs consuming (operating on) the abstraction.</span></span>

 <span data-ttu-id="455be-107">Uma abstração significativa e útil que é capaz de resistir ao teste de tempo é muito difícil de criar.</span><span class="sxs-lookup"><span data-stu-id="455be-107">A meaningful and useful abstraction that is able to withstand the test of time is very difficult to design.</span></span> <span data-ttu-id="455be-108">A dificuldade principal é obter o conjunto certo de membros, não há mais nem menos.</span><span class="sxs-lookup"><span data-stu-id="455be-108">The main difficulty is getting the right set of members, no more and no fewer.</span></span> <span data-ttu-id="455be-109">Se uma abstração tiver muitos membros, torna-se difícil ou até mesmo impossível de implementar.</span><span class="sxs-lookup"><span data-stu-id="455be-109">If an abstraction has too many members, it becomes difficult or even impossible to implement.</span></span> <span data-ttu-id="455be-110">Se ele tiver poucos membros para a funcionalidade prometida, ele se tornará inútil em muitos cenários interessantes.</span><span class="sxs-lookup"><span data-stu-id="455be-110">If it has too few members for the promised functionality, it becomes useless in many interesting scenarios.</span></span>

 <span data-ttu-id="455be-111">Muitas abstrações em uma estrutura também afetam negativamente a usabilidade da estrutura.</span><span class="sxs-lookup"><span data-stu-id="455be-111">Too many abstractions in a framework also negatively affect usability of the framework.</span></span> <span data-ttu-id="455be-112">Geralmente, é muito difícil entender uma abstração sem entender como ela se encaixa na visão mais ampla das implementações concretas e das APIs que operam na abstração.</span><span class="sxs-lookup"><span data-stu-id="455be-112">It is often quite difficult to understand an abstraction without understanding how it fits into the larger picture of the concrete implementations and the APIs operating on the abstraction.</span></span> <span data-ttu-id="455be-113">Além disso, os nomes de abstrações e seus membros são necessariamente abstratos, o que geralmente os torna confusos e inacessívels sem primeiro compreender o contexto mais amplo de seu uso.</span><span class="sxs-lookup"><span data-stu-id="455be-113">Also, names of abstractions and their members are necessarily abstract, which often makes them cryptic and unapproachable without first understanding the broader context of their usage.</span></span>

 <span data-ttu-id="455be-114">No entanto, as abstrações fornecem extensibilidade extremamente poderosa que os outros mecanismos de extensibilidade geralmente não podem corresponder.</span><span class="sxs-lookup"><span data-stu-id="455be-114">However, abstractions provide extremely powerful extensibility that the other extensibility mechanisms cannot often match.</span></span> <span data-ttu-id="455be-115">Eles estão no núcleo de muitos padrões arquitetônicos, como plug-ins, inversão de controle (IoC), pipelines e assim por diante.</span><span class="sxs-lookup"><span data-stu-id="455be-115">They are at the core of many architectural patterns, such as plug-ins, inversion of control (IoC), pipelines, and so on.</span></span> <span data-ttu-id="455be-116">Eles também são extremamente importantes para a capacidade de teste de estruturas.</span><span class="sxs-lookup"><span data-stu-id="455be-116">They are also extremely important for testability of frameworks.</span></span> <span data-ttu-id="455be-117">As boas abstrações possibilitam o stub de dependências pesadas para fins de teste de unidade.</span><span class="sxs-lookup"><span data-stu-id="455be-117">Good abstractions make it possible to stub out heavy dependencies for the purpose of unit testing.</span></span> <span data-ttu-id="455be-118">Em resumo, as abstrações são responsáveis pelas sofisticações buscadas das estruturas orientadas a objeto modernas.</span><span class="sxs-lookup"><span data-stu-id="455be-118">In summary, abstractions are responsible for the sought-after richness of the modern object-oriented frameworks.</span></span>

 <span data-ttu-id="455be-119">❌ não fornecem abstrações, a menos que sejam testadas desenvolvendo várias implementações concretas e APIs que consomem as abstrações.</span><span class="sxs-lookup"><span data-stu-id="455be-119">❌ DO NOT provide abstractions unless they are tested by developing several concrete implementations and APIs consuming the abstractions.</span></span>

 <span data-ttu-id="455be-120">✔️ Escolha cuidadosamente entre uma classe abstrata e uma interface ao projetar uma abstração.</span><span class="sxs-lookup"><span data-stu-id="455be-120">✔️ DO choose carefully between an abstract class and an interface when designing an abstraction.</span></span>

 <span data-ttu-id="455be-121">✔️ Considere fornecer testes de referência para implementações concretas de abstrações.</span><span class="sxs-lookup"><span data-stu-id="455be-121">✔️ CONSIDER providing reference tests for concrete implementations of abstractions.</span></span> <span data-ttu-id="455be-122">Esses testes devem permitir que os usuários testem se suas implementações implementam corretamente o contrato.</span><span class="sxs-lookup"><span data-stu-id="455be-122">Such tests should allow users to test whether their implementations correctly implement the contract.</span></span>
