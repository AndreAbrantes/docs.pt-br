 <span data-ttu-id="f3174-117"><xref:System.Collections.Concurrent.ConcurrentDictionary%602> foi projetado para cenários multi-threaded.</span><span class="sxs-lookup"><span data-stu-id="f3174-117"><xref:System.Collections.Concurrent.ConcurrentDictionary%602> is designed for multithreaded scenarios.</span></span> <span data-ttu-id="f3174-118">Você não precisa usar bloqueios em seu código para adicionar ou remover itens da coleção.</span><span class="sxs-lookup"><span data-stu-id="f3174-118">You do not have to use locks in your code to add or remove items from the collection.</span></span> <span data-ttu-id="f3174-119">No entanto, sempre é possível que um thread recupere um valor e que outro thread atualize imediatamente a coleção, dando um novo valor à mesma chave.</span><span class="sxs-lookup"><span data-stu-id="f3174-119">However, it is always possible for one thread to retrieve a value, and another thread to immediately update the collection by giving the same key a new value.</span></span>  
  
 <span data-ttu-id="f3174-120">Além disso, embora todos os métodos de <xref:System.Collections.Concurrent.ConcurrentDictionary%602> sejam thread-safe, nem todos os métodos são atômicos, especificamente <xref:System.Collections.Concurrent.ConcurrentDictionary%602.GetOrAdd%2A> e <xref:System.Collections.Concurrent.ConcurrentDictionary%602.AddOrUpdate%2A>.</span><span class="sxs-lookup"><span data-stu-id="f3174-120">Also, although all methods of <xref:System.Collections.Concurrent.ConcurrentDictionary%602> are thread-safe, not all methods are atomic, specifically <xref:System.Collections.Concurrent.ConcurrentDictionary%602.GetOrAdd%2A> and <xref:System.Collections.Concurrent.ConcurrentDictionary%602.AddOrUpdate%2A>.</span></span> <span data-ttu-id="f3174-121">O delegado do usuário que é passado para esses métodos é invocado fora do bloqueio interno do dicionário (isso é feito para impedir que um código desconhecido bloqueie todos os threads).</span><span class="sxs-lookup"><span data-stu-id="f3174-121">The user delegate that is passed to these methods is invoked outside of the dictionary's internal lock (this is done to prevent unknown code from blocking all threads).</span></span> <span data-ttu-id="f3174-122">Portanto, é possível que ocorra esta sequência de eventos:</span><span class="sxs-lookup"><span data-stu-id="f3174-122">Therefore, it is possible for this sequence of events to occur:</span></span>  
  
 <span data-ttu-id="f3174-123">1\) O threadA chama <xref:System.Collections.Concurrent.ConcurrentDictionary%602.GetOrAdd%2A>, não localiza nenhum item e, ao invocar o delegado valueFactory, cria um novo item para adicionar.</span><span class="sxs-lookup"><span data-stu-id="f3174-123">1\) threadA calls <xref:System.Collections.Concurrent.ConcurrentDictionary%602.GetOrAdd%2A>, finds no item and creates a new item to Add by invoking the valueFactory delegate.</span></span>  
  
 <span data-ttu-id="f3174-124">2\) O threadB chama <xref:System.Collections.Concurrent.ConcurrentDictionary%602.GetOrAdd%2A> simultaneamente, seu delegado valueFactory é invocado e chega ao bloqueio interno antes de threadA; assim, seu novo par chave-valor é adicionado ao dicionário.</span><span class="sxs-lookup"><span data-stu-id="f3174-124">2\) threadB calls <xref:System.Collections.Concurrent.ConcurrentDictionary%602.GetOrAdd%2A> concurrently, its valueFactory delegate is invoked and it arrives at the internal lock before threadA, and so its new key-value pair is added to the dictionary.</span></span>  
  
 <span data-ttu-id="f3174-125">3\) o delegado do usuário de threadA é concluído e o thread chega ao bloqueio, mas agora vê que o item já existe.</span><span class="sxs-lookup"><span data-stu-id="f3174-125">3\) threadA's user delegate completes, and the thread arrives at the lock, but now sees that the item exists already.</span></span>  
  
 <span data-ttu-id="f3174-126">4\) o threadA executa um "Get" e retorna os dados que já foram adicionados pelo threadB.</span><span class="sxs-lookup"><span data-stu-id="f3174-126">4\) threadA performs a "Get" and returns the data that was previously added by threadB.</span></span>
