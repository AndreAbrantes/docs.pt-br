<span data-ttu-id="f2227-103">Proteção estendida para autenticação ajuda a impedir ataques man-in-the-middle (MITM), em que um invasor intercepta as credenciais do cliente e encaminha-os para um servidor.</span><span class="sxs-lookup"><span data-stu-id="f2227-103">Extended Protection for Authentication helps protect against man-in-the-middle (MITM) attacks, in which an attacker intercepts a client’s credentials and forwards them to a server.</span></span>  
  
 <span data-ttu-id="f2227-104">Considere um cenário com três participantes: um invasor, cliente e servidor.</span><span class="sxs-lookup"><span data-stu-id="f2227-104">Consider a scenario with three participants: a client, server, and attacker.</span></span> <span data-ttu-id="f2227-105">O servidor tem a URL `https://server`, enquanto que o invasor terá a URL `https://attacker`.</span><span class="sxs-lookup"><span data-stu-id="f2227-105">The server has the URL `https://server`, whereas the attacker has the URL `https://attacker`.</span></span> <span data-ttu-id="f2227-106">O invasor truques do cliente para acessar o invasor, como se fosse o servidor.</span><span class="sxs-lookup"><span data-stu-id="f2227-106">The attacker tricks the client into accessing the attacker as if it were the server.</span></span> <span data-ttu-id="f2227-107">O invasor, em seguida, envia uma solicitação ao servidor.</span><span class="sxs-lookup"><span data-stu-id="f2227-107">The attacker then sends a request to the server.</span></span> <span data-ttu-id="f2227-108">Se o invasor está tentando acessar um recurso seguro, o servidor responde ao invasor com um cabeçalho WWW-Authenticate.</span><span class="sxs-lookup"><span data-stu-id="f2227-108">If the attacker is trying to access a secure resource, the server replies to the attacker with a WWW-Authenticate header.</span></span> <span data-ttu-id="f2227-109">O invasor não tem as informações de autenticação, portanto, ele envia o cabeçalho WWW-Authenticate logon no cliente.</span><span class="sxs-lookup"><span data-stu-id="f2227-109">The attacker does not have the authentication information, so it sends the WWW-Authenticate header on to the client.</span></span> <span data-ttu-id="f2227-110">O cliente envia o cabeçalho de autorização para o invasor, e o invasor envia o cabeçalho de logon no servidor e obtém acesso aos recursos de segurança usando as credenciais do cliente.</span><span class="sxs-lookup"><span data-stu-id="f2227-110">The client sends the Authorization header to the attacker, and the attacker sends the header on to the server and gets access to the secure resources using the client’s credentials.</span></span>  
  
 <span data-ttu-id="f2227-111">Atualmente, quando um aplicativo cliente autentica o servidor usando Kerberos, Digest ou NTLM usando HTTPS, primeiro é estabelecer um canal de segurança de nível de transporte (TLS) e a autenticação é feita usando esse canal.</span><span class="sxs-lookup"><span data-stu-id="f2227-111">Currently, when a client application authenticates itself to the server using Kerberos, Digest, or NTLM using HTTPS, a Transport Level Security (TLS) channel is first established and authentication takes place using this channel.</span></span> <span data-ttu-id="f2227-112">No entanto, não há nenhuma associação entre a chave de sessão gerada pelo protocolo (SSL) e a chave de sessão que é gerada durante a autenticação.</span><span class="sxs-lookup"><span data-stu-id="f2227-112">However, there is no binding between the session key generated by Secure Sockets Layer (SSL) and the session key that is generated during authentication.</span></span> <span data-ttu-id="f2227-113">Assim, no cenário anterior, se a comunicação ocorre por um TLS (como um canal HTTPS), existem dois canais SSL criados: um entre o cliente e o invasor e outra entre o invasor e o servidor.</span><span class="sxs-lookup"><span data-stu-id="f2227-113">So, in the previous scenario, if the communication takes places over a TLS (such as an HTTPS channel), there are two SSL channels created: one between the client and the attacker, and another between the attacker and the server.</span></span> <span data-ttu-id="f2227-114">As credenciais do cliente são enviadas do cliente para o servidor pela primeira vez no canal SSL entre o cliente e o invasor e, em seguida, o canal entre o invasor e o servidor.</span><span class="sxs-lookup"><span data-stu-id="f2227-114">The client’s credentials are sent from the client to the server first over the SSL channel between the client and the attacker and then over the channel between the attacker and the server.</span></span> <span data-ttu-id="f2227-115">Depois que as credenciais do cliente acessar o servidor, o servidor verifica as credenciais sem detectar que o canal em que essas credenciais foram enviadas originou-se com o invasor e não no cliente.</span><span class="sxs-lookup"><span data-stu-id="f2227-115">Once the client’s credentials reach the server, the server verifies the credentials without detecting that the channel over which those credentials were sent originated with the attacker and not the client.</span></span>  
  
 <span data-ttu-id="f2227-116">A solução é usar um canal externo protegida por TLS e um canal interno cliente autenticado e passar um canal de associação CBT (Token) para o servidor.</span><span class="sxs-lookup"><span data-stu-id="f2227-116">The solution is to use a TLS-secured outer channel and a client-authenticated inner channel, and to pass a Channel Binding Token (CBT) to the server.</span></span> <span data-ttu-id="f2227-117">Quando o CBT é uma propriedade do canal externa protegida por TLS e é usado para associar o canal externo para uma conversa no canal interno do cliente autenticado.</span><span class="sxs-lookup"><span data-stu-id="f2227-117">The CBT is a property of the TLS-secured outer channel, and is used to bind the outer channel to a conversation over the client-authenticated inner channel.</span></span>  
  
 <span data-ttu-id="f2227-118">No cenário anterior, quando o CBT do canal TLS cliente invasor é mesclado com as informações de autorização que são enviadas ao servidor.</span><span class="sxs-lookup"><span data-stu-id="f2227-118">In the previous scenario, the CBT of the client-attacker TLS channel is merged with the authorization information that is sent to the server.</span></span> <span data-ttu-id="f2227-119">Um servidor com reconhecimento de CBT compara o CBT contido nas informações de autenticação de cliente, que corresponde ao canal de invasor do cliente para o CBT anexado ao canal de servidor do invasor.</span><span class="sxs-lookup"><span data-stu-id="f2227-119">A CBT-aware server compares the CBT contained in the client authentication information, which corresponds to the client-attacker channel, to the CBT attached to the attacker-server channel.</span></span> <span data-ttu-id="f2227-120">Um CBT é específico para o destino do canal, para que o cliente-invasor CBT não coincide com o servidor de invasor CBT.</span><span class="sxs-lookup"><span data-stu-id="f2227-120">A CBT is specific to a channel’s destination, so the client-attacker CBT does not match the attacker-server CBT.</span></span> <span data-ttu-id="f2227-121">Isso permite que o servidor detecta o ataque MITM e recusar a solicitação de autenticação.</span><span class="sxs-lookup"><span data-stu-id="f2227-121">This lets the server detect the MITM attack and refuse the authentication request.</span></span>  
  
 <span data-ttu-id="f2227-122">O lado do cliente não requer qualquer configuração.</span><span class="sxs-lookup"><span data-stu-id="f2227-122">The client side does not require any configuration setting.</span></span> <span data-ttu-id="f2227-123">Depois que o cliente foi atualizado para passar o CBT para o servidor, ele sempre faz isso.</span><span class="sxs-lookup"><span data-stu-id="f2227-123">Once the client has been updated to pass the CBT to the server, it always does so.</span></span> <span data-ttu-id="f2227-124">Se o servidor também foi atualizado, ele pode ser configurado para usar o CBT ou ignorá-lo.</span><span class="sxs-lookup"><span data-stu-id="f2227-124">If the server has also been updated, it can be configured to use the CBT or ignore it.</span></span> <span data-ttu-id="f2227-125">Se não tiver sido atualizado, ela será ignorada.</span><span class="sxs-lookup"><span data-stu-id="f2227-125">If it has not been updated, it ignores it.</span></span>  
  
 <span data-ttu-id="f2227-126">O servidor pode ter os seguintes níveis de proteção:</span><span class="sxs-lookup"><span data-stu-id="f2227-126">The server can have the following levels of protection:</span></span>  
  
-   <span data-ttu-id="f2227-127">nenhuma.</span><span class="sxs-lookup"><span data-stu-id="f2227-127">None.</span></span> <span data-ttu-id="f2227-128">Nenhuma validação de associação de canal é executada.</span><span class="sxs-lookup"><span data-stu-id="f2227-128">No channel binding validation is performed.</span></span> <span data-ttu-id="f2227-129">Esse é o comportamento de todos os servidores que não foram atualizados.</span><span class="sxs-lookup"><span data-stu-id="f2227-129">This is the behavior of all servers that have not been updated.</span></span>  
  
-   <span data-ttu-id="f2227-130">Parcial.</span><span class="sxs-lookup"><span data-stu-id="f2227-130">Partial.</span></span> <span data-ttu-id="f2227-131">Todos os clientes que foram atualizados devem fornecer informações de associação de canal para o servidor.</span><span class="sxs-lookup"><span data-stu-id="f2227-131">All clients that have been updated must provide channel binding information to the server.</span></span> <span data-ttu-id="f2227-132">Clientes que não foram atualizados não precisa fazer isso.</span><span class="sxs-lookup"><span data-stu-id="f2227-132">Clients that have not been updated do not have to do so.</span></span> <span data-ttu-id="f2227-133">Isso é uma opção intermediária que permite a compatibilidade de aplicativos.</span><span class="sxs-lookup"><span data-stu-id="f2227-133">This is an intermediate option that allows for application compatibility.</span></span>  
  
-   <span data-ttu-id="f2227-134">Completo.</span><span class="sxs-lookup"><span data-stu-id="f2227-134">Full.</span></span> <span data-ttu-id="f2227-135">Todos os clientes devem fornecer informações de associação de canal.</span><span class="sxs-lookup"><span data-stu-id="f2227-135">All clients must provide channel binding information.</span></span> <span data-ttu-id="f2227-136">O servidor rejeita solicitações de autenticação de clientes que não faça isso.</span><span class="sxs-lookup"><span data-stu-id="f2227-136">The server rejects authentication requests from clients that do not do so.</span></span>
