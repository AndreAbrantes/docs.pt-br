## <a name="increasing-middle-tier-client-performance"></a><span data-ttu-id="cded5-104">Aumentando o desempenho do cliente de camada intermediária</span><span class="sxs-lookup"><span data-stu-id="cded5-104">Increasing Middle-Tier Client Performance</span></span>  
 <span data-ttu-id="cded5-105">Em comparação com as tecnologias de comunicações anteriores, como serviços Web usando [!INCLUDE[vstecasp](../../../../includes/vstecasp-md.md)], a criação de uma instância do cliente WCF pode ser mais complexa devido aos avançado conjunto de recursos do WCF.</span><span class="sxs-lookup"><span data-stu-id="cded5-105">Compared to previous communications technologies, such as Web services using [!INCLUDE[vstecasp](../../../../includes/vstecasp-md.md)], the creation of a WCF client instance can be more complex due to the rich feature set of WCF.</span></span> <span data-ttu-id="cded5-106">Por exemplo, quando um <xref:System.ServiceModel.ChannelFactory%601> objeto é aberto. ele pode estabelecer uma sessão segura com o serviço, um procedimento que aumenta o tempo de inicialização para a instância do cliente.</span><span class="sxs-lookup"><span data-stu-id="cded5-106">For example, when a <xref:System.ServiceModel.ChannelFactory%601> object is opened it can establish a secure session with the service, a procedure that increases the startup time for the client instance.</span></span> <span data-ttu-id="cded5-107">Normalmente, esses recursos adicionais não afetam aplicativos cliente significativamente desde que o cliente WCF faz várias chamadas e, em seguida, fecha.</span><span class="sxs-lookup"><span data-stu-id="cded5-107">Typically, these additional feature capabilities do not affect client applications greatly since the WCF client makes several calls, and then closes.</span></span>  
  
 <span data-ttu-id="cded5-108">Aplicativos cliente de camada intermediária, no entanto, podem criar muitos objetos de cliente WCF rapidamente e, como resultado, a experiência de requisitos de inicialização maior.</span><span class="sxs-lookup"><span data-stu-id="cded5-108">Middle-tier client applications, however, can create many WCF client objects quickly and, as a result, experience increased initialization requirements.</span></span> <span data-ttu-id="cded5-109">Há duas abordagens principais para aumentar o desempenho dos aplicativos de camada intermediária, ao chamar serviços:</span><span class="sxs-lookup"><span data-stu-id="cded5-109">There are two main approaches to increasing the performance of middle-tier applications when calling services:</span></span>  
  
- <span data-ttu-id="cded5-110">Armazenar em cache o objeto de cliente WCF e reutilizá-lo nas próximas chamadas sempre que possível.</span><span class="sxs-lookup"><span data-stu-id="cded5-110">Cache the WCF client object and reuse it for subsequent calls where possible.</span></span>  
  
- <span data-ttu-id="cded5-111">Criar um <xref:System.ServiceModel.ChannelFactory%601> de objeto e, em seguida, usar esse objeto para criar objetos de canal para cada chamada de novo cliente do WCF.</span><span class="sxs-lookup"><span data-stu-id="cded5-111">Create a <xref:System.ServiceModel.ChannelFactory%601> object and then use that object to create new WCF client channel objects for each call.</span></span>  
  
 <span data-ttu-id="cded5-112">Questões a serem consideradas ao usar esses métodos incluem:</span><span class="sxs-lookup"><span data-stu-id="cded5-112">Issues to consider when using these approaches include:</span></span>  
  
- <span data-ttu-id="cded5-113">Se o serviço mantém um estado específico do cliente por meio de uma sessão, é possível reutilizar o cliente do WCF de camada intermediária com solicitações de cliente de várias camadas porque o estado do serviço está ligado do cliente de camada intermediária.</span><span class="sxs-lookup"><span data-stu-id="cded5-113">If the service is maintaining a client-specific state by using a session, then you cannot reuse the middle-tier WCF client with multiple-client tier requests because the service's state is tied to that of the middle-tier client.</span></span>  
  
- <span data-ttu-id="cded5-114">Se o serviço deve executar a autenticação em uma base por cliente, você deve criar um novo cliente para cada solicitação de entrada na camada intermediária em vez de reutilizar o cliente do WCF de camada intermediária (ou o objeto de canal de cliente do WCF), porque as credenciais do cliente de camada intermediária não pode ser modificado depois que o cliente do WCF (ou <xref:System.ServiceModel.ChannelFactory%601>) foi criado.</span><span class="sxs-lookup"><span data-stu-id="cded5-114">If the service must perform authentication on a per-client basis, you must create a new client for each incoming request on the middle tier instead of reusing the middle-tier WCF client (or WCF client channel object) because the client credentials of the middle tier cannot be modified after the WCF client (or <xref:System.ServiceModel.ChannelFactory%601>) has been created.</span></span>  
  
- <span data-ttu-id="cded5-115">Enquanto os clientes criados pelos canais e canais são thread-safe, eles podem não dar suporte a gravação de mais de uma mensagem para a transmissão simultaneamente.</span><span class="sxs-lookup"><span data-stu-id="cded5-115">While channels and clients created by the channels are thread-safe, they might not support writing more than one message to the wire concurrently.</span></span> <span data-ttu-id="cded5-116">Se você estiver enviando mensagens grandes, especialmente se o streaming, a operação de envio pode bloquear a espera para outro enviar concluir.</span><span class="sxs-lookup"><span data-stu-id="cded5-116">If you are sending large messages, particularly if streaming, the send operation might block waiting for another send to complete.</span></span> <span data-ttu-id="cded5-117">Isso faz com que dois tipos de problemas: uma falta de simultaneidade e a possibilidade de deadlock se o fluxo de controle retorna para o serviço reutilizando o canal (ou seja, o cliente compartilhado chama um serviço cujos resultados de caminho de código em um retorno de chamada para o cliente compartilhado).</span><span class="sxs-lookup"><span data-stu-id="cded5-117">This causes two sorts of problems: a lack of concurrency and the possibility of deadlock if the flow of control returns to the service reusing the channel (that is, the shared client calls a service whose code path results in a callback to the shared client).</span></span> <span data-ttu-id="cded5-118">Isso é verdadeiro independentemente do tipo de cliente do WCF que é reutilizar.</span><span class="sxs-lookup"><span data-stu-id="cded5-118">This is true regardless of the type of WCF client you reuse.</span></span>  
  
- <span data-ttu-id="cded5-119">Você deve tratar os canais com falha, independentemente se você compartilhar o canal.</span><span class="sxs-lookup"><span data-stu-id="cded5-119">You must handle faulted channels regardless of whether you share the channel.</span></span> <span data-ttu-id="cded5-120">Quando são reutilizados e canais, no entanto, um canal com falha pode levar mais de uma solicitação pendente ou enviar.</span><span class="sxs-lookup"><span data-stu-id="cded5-120">When channels are reused, however, a faulting channel can take down more than one pending request or send.</span></span>  
  
 <span data-ttu-id="cded5-121">Para obter um exemplo que demonstra as práticas recomendadas para reutilização de um cliente para várias solicitações, consulte [vinculação de dados em um cliente do ASP.NET](../../../../docs/framework/wcf/samples/data-binding-in-an-aspnet-client.md).</span><span class="sxs-lookup"><span data-stu-id="cded5-121">For an example that demonstrates best practices for reusing a client for multiple requests, see [Data Binding in an ASP.NET Client](../../../../docs/framework/wcf/samples/data-binding-in-an-aspnet-client.md).</span></span>
