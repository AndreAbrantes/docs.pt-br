# <a name="federation-and-trust"></a><span data-ttu-id="fc46a-102">Federação e confiabilidade</span><span class="sxs-lookup"><span data-stu-id="fc46a-102">Federation and Trust</span></span>

## <a name="services-security-token-services-and-trust"></a><span data-ttu-id="fc46a-104">Serviços, serviços de Token de segurança e confiança</span><span class="sxs-lookup"><span data-stu-id="fc46a-104">Services, Security Token Services, and Trust</span></span>  
 <span data-ttu-id="fc46a-105">Serviços que expõem pontos de extremidade federados normalmente esperam que os clientes se autentiquem usando um token fornecido por um emissor específico.</span><span class="sxs-lookup"><span data-stu-id="fc46a-105">Services that expose federated endpoints typically expect clients to authenticate using a token provided by a specific issuer.</span></span> <span data-ttu-id="fc46a-106">É importante que o serviço está configurado com as credenciais corretas para o emissor; Caso contrário, ele não poderá verificar assinaturas em tokens emitidos e o cliente será capaz de se comunicar com o serviço.</span><span class="sxs-lookup"><span data-stu-id="fc46a-106">It is important that the service is configured with the correct credentials for the issuer; otherwise, it will not be able to verify signatures over the issued tokens, and the client will be unable to communicate with the service.</span></span> <span data-ttu-id="fc46a-107">Para obter mais informações sobre como configurar as credenciais do emissor no serviço, consulte [como: Configurar credenciais em um serviço de Federação](../../../../docs/framework/wcf/feature-details/how-to-configure-credentials-on-a-federation-service.md).</span><span class="sxs-lookup"><span data-stu-id="fc46a-107">For more information about configuring issuer credentials on the service, see [How to: Configure Credentials on a Federation Service](../../../../docs/framework/wcf/feature-details/how-to-configure-credentials-on-a-federation-service.md).</span></span>  
  
 <span data-ttu-id="fc46a-108">Da mesma forma, ao usar chaves simétricas, as chaves são criptografadas para o serviço de destino, portanto, você deve configurar o serviço de token de segurança com as credenciais corretas para o serviço de destino; Caso contrário, ele será não é possível criptografar a chave para o serviço de destino, e novamente, o cliente será capaz de se comunicar com o serviço.</span><span class="sxs-lookup"><span data-stu-id="fc46a-108">Similarly, when using symmetric keys, the keys are encrypted for the target service, so you must configure the security token service with the correct credentials for the target service; otherwise, it will be unable to encrypt the key for the target service, and again, the client will be unable to communicate with the service.</span></span>  
  
 <span data-ttu-id="fc46a-109">Os serviços WCF usam o valor da <xref:System.ServiceModel.Channels.LocalServiceSecuritySettings.MaxClockSkew%2A> propriedade no [SecurityBindingElement](../../../../docs/framework/wcf/diagnostics/wmi/securitybindingelement.md) para permitir o relógio distorção entre o cliente e o serviço.</span><span class="sxs-lookup"><span data-stu-id="fc46a-109">WCF services use the value of the <xref:System.ServiceModel.Channels.LocalServiceSecuritySettings.MaxClockSkew%2A> property on the [SecurityBindingElement](../../../../docs/framework/wcf/diagnostics/wmi/securitybindingelement.md) to allow for clock skew between the client and service.</span></span> <span data-ttu-id="fc46a-110">Na federação, o `MaxClockSkew` configuração se aplica ao relógio distorcer entre o cliente e o serviço de token de segurança de onde o cliente obteve o token emitido.</span><span class="sxs-lookup"><span data-stu-id="fc46a-110">In federation, the `MaxClockSkew` setting applies to clock skews between both the client and the security token service from where the client obtained the issued token.</span></span> <span data-ttu-id="fc46a-111">Portanto, os serviços de token de segurança não precisam fazer concessões de distorção de relógio ao definir tempos de efetivação e expiração do token emitido.</span><span class="sxs-lookup"><span data-stu-id="fc46a-111">Therefore, security token services need not make clock-skew allowances when setting the issued token's effective and expiration times.</span></span>  
  
> [!NOTE]
>  <span data-ttu-id="fc46a-112">A importância de defasagem horária aumenta à medida que diminui o tempo de vida do token emitido.</span><span class="sxs-lookup"><span data-stu-id="fc46a-112">The importance of clock skew increases as the lifetime of the issued token shortens.</span></span> <span data-ttu-id="fc46a-113">Na maioria dos casos, distorção de relógio não é um problema significativo se o tempo de vida de token é 30 minutos ou mais.</span><span class="sxs-lookup"><span data-stu-id="fc46a-113">In most cases, clock skew is not a significant issue if the token lifetime is 30 minutes or more.</span></span> <span data-ttu-id="fc46a-114">Cenários com tempos de vida mais curtos ou em que o tempo exato de validade do token é importante devem ser projetados para levar o distorção do relógio em conta.</span><span class="sxs-lookup"><span data-stu-id="fc46a-114">Scenarios with shorter lifetimes or where the exact validity time of the token is important should be designed to take clock skew into account.</span></span>

 <span data-ttu-id="fc46a-119">Por exemplo, o tempo limite no canal do cliente é definido como 30 segundos.</span><span class="sxs-lookup"><span data-stu-id="fc46a-119">For example, the time-out on <xref:System.ServiceModel.Channels.Binding.SendTimeout%2A> property (or both) need to be set to a value large enough to include the time taken to retrieve all issued tokens.</span></span>
